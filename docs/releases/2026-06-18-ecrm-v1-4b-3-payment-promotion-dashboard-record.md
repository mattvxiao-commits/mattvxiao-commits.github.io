# ECRM V1.4b-3 支付方式与活动效果仪表盘交付记录

## 1. 版本定位

V1.4b-3 在 V1.4b-2 SPU 表现仪表盘基础上增加支付方式统计和活动效果统计，让现场售卖后可以快速复盘收款结构、优惠加购贡献和满赠触发情况。

本版本继续保持轻量仪表盘策略，只做固定高价值统计，不做可配置图表、拖拽仪表盘或复杂分析系统。

## 2. 已完成内容

- 新增支付方式统计：
  - 微信订单数与金额。
  - 支付宝订单数与金额。
  - 现金订单数与金额。
  - 其他订单数与金额。
  - 未记录支付方式订单数与金额。
- 新增活动效果统计：
  - 优惠加购件数。
  - 优惠让利金额。
  - 参与优惠订单数。
  - 满赠触发订单数。
- 新增满赠档位触发统计：
  - 按满赠门槛金额聚合触发订单数。
  - 按门槛金额升序展示。
- 仪表盘页面新增：
  - “支付方式”模块。
  - “活动效果”指标条。
  - “满赠触发”模块。

## 3. 统计口径

### 3.1 范围

- 所有新增指标只统计当前时间范围内的已支付订单。
- 时间范围沿用 V1.4a：今日、昨天、近 3 天、近 7 天、自定义日期范围。
- 作废订单、待支付订单和范围外订单不计入本版本新增统计。

### 3.2 支付方式

- 支付方式金额使用订单 `payableAmount`。
- 支付方式金额不扣退款。
- 退款拆分回支付方式留给后续版本。
- `paymentMethod` 缺失的历史订单归入“未记录”。
- 页面固定展示五类支付方式：微信、支付宝、现金、其他、未记录。
- 即使某一类当前范围为 0 单，也保留该行，方便查看完整收款方式分布。

### 3.3 活动效果

- 优惠加购只统计订单明细中的 `discount_addon` 行。
- 优惠加购件数为 `discount_addon.quantity` 合计。
- 优惠让利金额为 `(originalUnitPrice - finalUnitPrice) * quantity` 合计。
- 参与优惠订单数按有 `discount_addon` 明细的订单去重。
- 满赠订单数按当前范围已支付订单中 `triggeredGiftTier` 为数字的订单数统计。

### 3.4 满赠触发

- 满赠触发按 `Order.triggeredGiftTier` 聚合。
- 只统计当前范围内已支付订单。
- 聚合结果按门槛金额升序展示。

## 6. 已知限制

- 本版本不把退款金额拆分回支付方式。
- 本版本不接入真实微信或支付宝支付 API。
- 本版本不做支付平台到账核验。
- 本版本不做库存风险 V1.4b-4。
- 本版本不做毛利、毛利率、成本快照或库存价值 V1.5。
- 本版本不做 CSV 导出。
- 本版本不做可配置图表或拖拽仪表盘。

## 7. 后续建议

V1.4b-4 建议继续推进库存风险统计，优先展示售罄 SKU、高风险 SKU、滞销 SKU 和补货建议，用于收摊后快速补货和下次出摊备货。

V1.5 再考虑成本快照、毛利、毛利率和库存价值。该部分需要订单写入时保存成本快照，复杂度高于 V1.4b，不建议混入当前小版本。
