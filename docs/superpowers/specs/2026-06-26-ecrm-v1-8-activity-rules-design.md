# ECRM V1.8 运营活动与促销规则模块化设计方案

## 1. 背景

V1.7.0 已完成线上发布和跨设备验收。当前系统已经支持固定加购优惠、固定满赠、运营赠礼、人工赠送、其他非销售出库、订单统计口径修正、数据页复盘和订单 Excel 导出。

这些能力解决了第一阶段真实摆摊的核心问题，但促销和运营活动仍然有一个明显限制：规则写死在系统配置里，无法适配更多摊主、更多活动方案和更多商品结构。

V1.8 的目标是把固定规则升级为“活动模板 + 规则引擎”。用户不需要写公式，只需要选择活动模板、设置触发条件、选择作用范围、配置叠加方式和成本归属。系统负责计算购物车价格、赠品、库存扣减、订单快照、仪表盘统计和导出字段。

## 2. 阶段目标

V1.8 要解决以下问题：

1. 让满赠、加购、折扣、运营赠礼从固定配置变成可配置活动。
2. 用统一规则引擎计算购物车命中结果，减少各页面各算各的风险。
3. 保存活动订单快照，确保历史订单不受后续活动配置修改影响。
4. 兼容 V1.7.0 及更早版本的历史订单。
5. 让订单记录、订单详情、统计口径修正、仪表盘、Excel 导出、JSON 备份都能识别新活动结构。

V1.8 不做自由公式编辑器，不做复杂会员系统，不接入社媒平台自动验证，不接入微信 / 支付宝 API，不做云同步。

## 3. 关键业务规则

### 3.1 同类活动不叠加

同一种活动模板内部默认不叠加。

满额赠品如果配置多个档位，只按最高命中档位生效。例如满 148 元时，只生效 148 档，不再叠加 35 档和 68 档。

加购优惠同理。同一条加购活动只按自身上限、优惠价和恢复原价规则计算，不重复套用多次。

### 3.2 不同活动默认不叠加

不同活动类型之间默认不可叠加。每个活动配置里提供一个开关：

```text
允许与其他活动叠加
```

默认关闭。

只有两个活动都开启“允许与其他活动叠加”时，才可以同时生效。只要任意一方关闭叠加，则按优先级只生效一个。

互斥时按以下规则选择：

1. 优先级高的活动生效。
2. 优先级相同，按活动列表顺序靠前的活动生效。

V1.8 第一版不做“自动选择优惠最大”的模式。满赠、折扣、加购、赠礼之间很难用一个简单公式比较优惠力度，过早加入会增加错误和解释成本。

### 3.3 成本归属

成本归属沿用 V1.6a 和 V1.6.3b 的经营复盘口径。

| 类型 | 成本归属 | 说明 |
|---|---|---|
| 正常销售 | 销售成本 | 有收入的商品明细成本 |
| 加购优惠销售 | 销售成本 + 优惠让利 | 商品成本归销售成本，原价与成交价差额计入优惠让利 |
| 满赠赠品 | 满赠成本 | 属于运营活动成本 |
| 运营赠礼 | 运营赠礼成本 | 属于运营活动成本 |
| 人工赠送 | 人工赠送成本 | 属于非经营出库成本 |
| 其他非销售出库 | 其他非销售出库成本 | 属于非经营出库成本 |

仪表盘需要同时支持拆分和合并：

- 销售成本。
- 满赠成本。
- 加购优惠让利。
- 运营赠礼成本。
- 人工赠送成本。
- 其他非销售出库成本。
- 销售成本 + 运营活动成本。
- 全出库成本。

### 3.4 活动作用范围

活动作用范围使用两级选择。

第一级选择范围类型：

```text
指定系列
指定 SPU
指定 SKU
```

第二级根据范围类型加载可选内容：

- 指定系列：显示已有商品系列。
- 指定 SPU：显示已有 SPU。
- 指定 SKU：显示具体商品 SKU。

如果商品没有填写系列，不进入“指定系列”候选项。后续如有需要，可以单独增加“未设置系列”范围。

运营赠礼、满赠这类活动如果只允许赠品，应只显示“可作为赠品”的候选范围。人工赠送和其他非销售出库不属于官方运营活动，可以选择全部商品。

### 3.5 订单快照

新活动不能只依赖当前设置。订单保存时必须写入活动快照。

订单活动快照至少包含：

- 活动 ID。
- 活动名称。
- 活动模板类型。
- 触发条件摘要。
- 作用范围。
- 是否允许叠加。
- 实际生效原因。
- 优惠金额。
- 赠品来源。
- 成本归属。
- 下单时售价。
- 下单时成交价。
- 下单时成本价。

后续修改、停用或删除活动配置，不应影响历史订单。

## 4. 活动模板

### 4.1 满额赠品

适用场景：满 35、满 68、满 148 等金额档位赠品。

用户可配置：

- 活动名称。
- 是否启用。
- 是否允许与其他活动叠加。
- 活动优先级。
- 一个或多个金额档位。
- 每个档位的赠品范围：系列 / SPU / SKU。
- 每个档位的赠送数量。
- 赠品库存不足时的提示。
- 成本归属，默认满赠成本。

规则：

- 同一满赠活动内只取最高命中档位。
- 满赠赠品不计销售收入。
- 满赠赠品扣库存、进订单明细、进库存流水。
- 满赠成本计入运营活动成本。

### 4.2 加购优惠

适用场景：任意消费后，指定 SPU 前几个按优惠价加购。

用户可配置：

- 活动名称。
- 是否启用。
- 是否允许与其他活动叠加。
- 活动优先级。
- 触发条件：任意消费后、满指定金额后、购买指定范围后。
- 加购范围：系列 / SPU / SKU。
- 优惠价。
- 每单最多优惠数量。
- 超出优惠数量后是否恢复原价。
- 只买加购范围商品时，是否要求第 1 件原价。

当前真实规则应可用新模板表达：

```text
任意消费后，指定 SPU 商品前 3 个优惠价。
如果只买该 SPU，第 1 个原价，第 2-4 个优惠，第 5 个恢复原价。
```

加购优惠商品仍是销售明细，计销售收入和销售成本。原价与成交价之间的差额计入优惠让利。

### 4.3 折扣优惠

适用场景：满额折扣、指定 SPU 折扣、指定系列折扣。

V1.8 第一版只做简单折扣，不做复杂公式。

用户可配置：

- 活动名称。
- 是否启用。
- 是否允许与其他活动叠加。
- 活动优先级。
- 触发条件：满额、购买指定范围、无门槛。
- 折扣范围：整单、指定系列、指定 SPU、指定 SKU。
- 折扣方式：百分比折扣、立减金额。

第一版不做阶梯折扣、不做满减叠加券、不做“第 N 件半价”。这些可以留到后续版本。

### 4.4 运营赠礼

适用场景：关注社媒、加入社群、现场互动、打卡、抽奖等活动赠礼。

用户可配置：

- 活动名称。
- 是否启用。
- 是否要求购物车内有正常销售商品。
- 赠礼范围：系列 / SPU / SKU。
- 默认赠送数量。
- 备注模板。
- 成本归属，默认运营赠礼成本。

运营赠礼默认不计销售收入，但要扣库存、进订单明细、进库存流水，并进入仪表盘活动成本统计。

备注不应写死为“关注小红书赠礼”。活动名称和备注模板由用户配置，例如：

- 关注小红书。
- 关注微博。
- 加入社群。
- 现场互动。
- 打卡赠礼。

### 4.5 人工赠送与其他非销售出库

人工赠送和其他非销售出库继续保留，但不归入官方运营活动模板。

规则：

- 可选择全部商品。
- 单价自动为 0。
- 不计销售收入。
- 不计销售毛利。
- 扣库存。
- 记录库存流水。
- 成本归入非经营出库成本。
- 必须填写原因或备注。

## 5. 旧订单兼容

V1.8 必须兼容 V1.7.0 及更早版本订单。

原则：

1. 历史订单不回算。
2. 新活动配置不影响历史订单。
3. 历史订单继续使用下单时保存的明细、价格、成本、赠品和统计口径。
4. 没有新活动 ID 的历史订单，可在展示层标记为“历史固定活动”。
5. 读取历史订单时做归一化，不强制改写原始订单数据。

建议做一个兼容层：

```text
读取 V1.8 新订单：使用 activitySnapshot。
读取 V1.7 或更早订单：转换为 legacy activity snapshot。
```

旧订单在仪表盘和导出中可以使用统一结构参与统计，但不因为用户修改新活动配置而变化。

## 6. 必须联动的模块

### 6.1 设置页

设置页需要新增活动管理区域：

- 活动列表。
- 新增活动。
- 编辑活动。
- 复制活动。
- 启用 / 停用活动。
- 活动模板选择。
- 作用范围选择。
- 叠加开关。
- 成本归属。
- 优先级 / 排序。

旧版促销配置需要迁移到新活动结构。导入旧备份时，如果缺少活动配置，应生成等价默认活动。

### 6.2 购物车

购物车必须使用统一规则引擎计算活动命中。

需要展示：

- 已命中的活动。
- 未满足条件但接近触发的活动提示。
- 优惠金额。
- 赠品。
- 活动互斥时实际生效的活动。
- 活动叠加时多个活动来源。

购物车中运营赠礼、人工赠送、其他出库仍要保持清晰，不应被折扣或满赠提示挤压。

### 6.3 收款页

收款页用于最后复核。

需要展示：

- 本单活动命中。
- 优惠金额。
- 赠品和库存提示。
- 成本归属摘要。
- 应收金额。

确认收款时，订单和订单明细必须写入活动快照。

### 6.4 订单记录

订单记录需要适配新活动结构：

- 支持按活动类型筛选。
- 支持按订单性质筛选。
- 支持识别销售 + 活动赠礼的混合订单。
- 支持显示历史固定活动标签。
- 订单性质标签继续基于明细归一化结果派生。

### 6.5 订单详情与统计口径修正

订单详情需要显示每个明细的活动来源和成本归属。

统计口径修正继续支持：

- 整单修正。
- 单商品修正。
- 正常销售。
- 运营赠礼。
- 人工赠送。
- 其他非销售出库。

修正统计口径不改变实收金额。涉及退款、作废、收入调整时，仍走退款或作废流程。

### 6.6 数据页

数据页需要适配活动模块：

- 活动筛选。
- 活动类型筛选。
- 正常销售、全部活动、运营赠礼、人工赠送、其他出库等既有口径继续保留。
- 新增按活动名称查看成本和效果。
- 区分满赠成本、加购优惠让利、运营赠礼成本、人工赠送成本、其他出库成本。
- 旧版活动订单按“历史固定活动”兼容展示。

### 6.7 Excel 导出

订单 Excel 需要增加活动字段：

- 活动 ID。
- 活动名称。
- 活动模板。
- 活动类型。
- 成本归属。
- 是否历史固定活动。
- 是否叠加。
- 优惠金额。
- 赠品来源。

旧订单导出时，活动 ID 可以为空，但不能丢失明细类型、成本和统计口径。

### 6.8 JSON 备份与导入

JSON 备份需要包含：

- 活动配置。
- 活动排序。
- 活动启用状态。
- 订单活动快照。

导入旧备份时：

- 如果没有活动配置，生成默认活动结构。
- 如果订单没有活动快照，读取时通过兼容层归一化。
- 不因为缺少新字段导致导入失败。

## 7. 数据模型方向

建议新增或扩展以下结构。

### 7.1 `Activity`

```ts
type Activity = {
  id: string;
  name: string;
  template: "tier_gift" | "addon_discount" | "discount" | "campaign_gift";
  enabled: boolean;
  priority: number;
  stackable: boolean;
  scope: ActivityScope;
  condition: ActivityCondition;
  benefit: ActivityBenefit;
  accountingCategory: ActivityAccountingCategory;
  createdAt: string;
  updatedAt: string;
};
```

### 7.2 `ActivityScope`

```ts
type ActivityScope = {
  targetType: "series" | "spu" | "sku";
  targetIds: string[];
  giftEligibleOnly?: boolean;
};
```

系列和 SPU 当前未必有独立 ID。第一版可以用稳定名称作为目标值，但需要在文档和代码中明确：改名会影响后续活动范围。后续如果要彻底解决，应引入独立的系列 ID 和 SPU ID。

### 7.3 `ActivitySnapshot`

```ts
type ActivitySnapshot = {
  activityId?: string;
  activityName: string;
  template: Activity["template"] | "legacy_fixed";
  scopeLabel: string;
  conditionLabel: string;
  stackable: boolean;
  accountingCategory: ActivityAccountingCategory;
  discountAmount: number;
  isLegacy: boolean;
};
```

### 7.4 `ActivityEvaluationResult`

```ts
type ActivityEvaluationResult = {
  appliedActivities: ActivitySnapshot[];
  blockedActivities: ActivitySnapshot[];
  discountTotal: number;
  giftLines: EvaluatedGiftLine[];
  linePriceAdjustments: EvaluatedLinePriceAdjustment[];
  warnings: string[];
};
```

这类结构只作为方向，最终命名应按现有领域模型调整。

## 8. 实施拆分

### 8.1 V1.8a：活动模型与规则引擎

目标：

- 新增活动模型。
- 新增规则引擎。
- 把当前固定满赠、加购、运营赠礼转换为默认活动。
- 保证购物车现有行为不倒退。
- 新增旧订单兼容读取层。

V1.8a 不做复杂活动配置 UI。可以先保留现有设置入口，但底层开始使用活动模型。

### 8.2 V1.8b：活动配置 UI

目标：

- 设置页新增活动管理。
- 支持新增、编辑、复制、启用、停用活动。
- 支持范围类型和范围内容两级选择。
- 支持叠加开关、优先级、成本归属。
- 支持活动模板表单。

### 8.3 V1.8c：全链路联动

目标：

- 订单保存活动快照。
- 订单记录支持活动筛选。
- 订单详情显示活动来源。
- 统计口径修正适配活动字段。
- 数据页支持活动成本和活动效果统计。
- Excel 导出增加活动字段。
- JSON 备份导入兼容新旧结构。

## 9. 验收标准

V1.8 完成后应满足：

1. 旧订单仍可查看、筛选、导出和统计。
2. 修改新活动配置不会改变旧订单复盘结果。
3. 当前固定满赠和加购规则可由新活动模型表达。
4. 购物车计算结果与 V1.7.0 已验收行为一致，除非用户主动修改活动配置。
5. 用户可以配置满赠、加购、简单折扣和运营赠礼。
6. 活动默认不叠加，只有双方都允许叠加时才同时生效。
7. 活动范围可以按系列、SPU、SKU 选择。
8. 订单明细保存活动快照和成本归属。
9. 数据页能拆分销售成本、活动成本和非经营出库成本。
10. Excel 导出和 JSON 备份不丢活动信息。

## 10. 后续 UI/UE 关系

V1.8 会改动设置页、购物车、收款页、订单详情、数据页和导出说明。V1.7.0 验收后记录的 UI/UE 规范问题不建议在 V1.8 前单独精修。

推荐顺序：

1. 先完成 V1.8 活动模块。
2. 再做全局 UI/UE 规格统一审查。
3. 最后做移动端专项适配。

这样可以减少重复调整，也能让 UI 规范建立在稳定业务结构之上。
