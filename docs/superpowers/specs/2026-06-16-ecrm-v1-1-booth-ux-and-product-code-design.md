# ECRM V1.1 摊位操作体验与商品编码设计规格

日期：2026-06-16

## 1. 背景

当前 MVP 已完成商品管理、售卖开单、购物车、促销、收款二维码、订单保存、库存扣减、基础仪表盘和备份能力。用户测试后确认多商品添加和图片显示基本可用，但现场售卖页、购物车、整体界面密度、促销配置和商品编码仍不够适合真实摆摊场景。

V1.1 的目标不是扩展成完整 CRM，而是把现有 MVP 调整成更像“摊位现场 POS 工具”的形态：一屏能看到更多商品，购物车不长期占用商品区，商品编码更适合管理，促销和赠品配置减少手填错误。

## 2. 已确认问题

### 2.1 商品唯一 ID 未呈现

当前系统已有内部商品 ID，用于数据库、购物车、订单和库存记录关联，但该 ID 是技术标识，不适合摊主现场识别商品。

当前商品没有独立的用户可读 SKU 编码字段，也没有完整商品编码，因此后续按具体 SKU 配置赠品、搜索商品、核对库存和订单时不够清晰。

### 2.2 售卖页商品展示过于卡片化

当前售卖页只有图片卡片/网格展示。该形式适合靠图片识别商品，但商品多时空间利用率低，现场快速开单效率不足。

### 2.3 购物车固定占据右侧区域

当前购物车默认固定占据页面右侧，无法收起或展开。桌面和 iPad 横屏下商品列表被持续压缩，小屏或 iPad 竖屏下操作空间更紧张。

购物车商品行目前缺少小尺寸缩略图，核对已加入商品时不够直观。

### 2.4 整体 GUI 密度偏低

当前页面标题、文字比例、卡片间距和留白偏展示化，不够适合现场 POS 高频操作。用户需要更紧凑的信息布局，而不是营销页或展示型后台风格。

### 2.5 优惠 SPU 配置依赖手填

当前加购优惠底层已经按 SPU 计算，但设置页仍要求手动输入优惠 SPU。手填容易出现空格、错字、大小写不一致等问题。

### 2.6 满赠只能配置具体商品

当前赠品档位只能选择具体商品/SKU。用户希望支持：

- 指定某个 SPU 下所有 SKU 作为赠品池。
- 或指定某个具体 SKU 作为赠品。

这会影响促销配置、购物车展示、结账确认、订单记录和库存扣减，不是单纯的下拉菜单调整。

## 3. V1.1 总体策略

采用分段实施：

### 3.1 V1.1a：现场操作效率修复

优先修复现场使用体验，不改变核心订单数据模型。

范围：

- 售卖页增加“紧凑列表 / 图片网格”视图切换。
- 默认进入紧凑列表。
- 购物车改为可展开/收起抽屉。
- 购物车商品行增加缩略图。
- 全局 UI 密度压缩，重点优化售卖页、商品管理页、设置页。
- 优惠 SPU 改为从商品库自动识别并下拉选择。

### 3.2 V1.1b：商品编码与赠品规则增强

在 V1.1a 稳定后进行数据模型增强。

范围：

- 商品新增 SPU 编码、SKU 编码、完整商品编码。
- 完整商品编码由系统自动生成。
- 商品管理、售卖页、购物车、设置页展示完整商品编码。
- 赠品配置支持指定 SPU 或指定 SKU。
- 指定 SPU 赠品在结账/确认收款前必须手动选择实际赠出的 SKU。
- 订单和库存按实际赠出的 SKU 记录和扣减。

### 3.3 不采用的一次性方案

不建议一次性同时修改 UI、商品编码、赠品配置、订单保存和库存扣减。原因是测试面过大，且用户当前有较强的近期可用性要求。V1.1 应保证每个阶段都能真实落地和回归验证。

## 4. V1.1a 详细设计

### 4.1 售卖页视图切换

售卖页增加视图切换控件：

- 紧凑列表。
- 图片网格。

默认视图为“紧凑列表”。

紧凑列表每行展示：

- 商品缩略图。
- 商品名称。
- SPU / 商品分组。
- V1.1a 暂不展示商品编码；V1.1b 后展示完整商品编码。
- 售价。
- 库存。
- 当前购物车内数量。
- 加购按钮。

图片网格保留现有图片识别优势，但应压缩卡片尺寸、标题字号、间距和图片区域高度，避免一屏只能看到少量商品。

视图切换状态可以先保存在组件状态中。若实现成本低，可持久化到浏览器本地设置，使用户下次打开仍保持上次选择。

### 4.2 商品筛选

保留当前 SPU 筛选。

筛选按钮应更紧凑，适合横向滚动：

- 全部。
- 每个已存在 SPU。

SPU 来源为当前商品库中有效商品的去重结果。

V1.1a 不强制新增搜索，但如果 UI 调整过程中成本较低，可加入按商品名称和 SPU 的关键词搜索。搜索不是 V1.1a 的硬性完成项。

### 4.3 购物车抽屉

购物车从固定右侧面板调整为可收起/展开抽屉。

桌面、笔记本、iPad 横屏：

- 默认可收起。
- 展开时从右侧滑出或占据右侧固定宽度。
- 收起时商品列表占据完整主区域。
- 右下角保留悬浮购物车按钮，显示件数和应收金额。

iPad 竖屏和窄屏：

- 购物车可作为底部抽屉或近似全屏面板打开。
- 打开后应保留明确关闭/返回入口。

购物车打开状态规则：

- 点击悬浮购物车按钮：打开购物车。
- 点击购物车关闭按钮：收起购物车，返回商品列表。
- 点击“暂存”：保持当前暂存逻辑，并收起购物车返回商品列表。
- 点击“下单”：进入收款/结账子页面。
- 从结账返回购物车时，应回到购物车展开状态。

### 4.4 购物车商品缩略图

购物车每个非赠品/赠品商品行均展示小尺寸商品缩略图：

- 有图片：保持原图比例，以 contain 或 cover 的方式显示在固定尺寸缩略图框中，优先不裁切关键信息。
- 无图片：显示简短占位符，例如商品首字或默认占位。

缩略图尺寸应稳定，不因图片比例或文字长度改变购物车行高。

### 4.5 UI 密度压缩

V1.1a 采用“摊位紧凑模式”作为设计基准。

核心原则：

- 页面标题不再使用超大展示字号。
- 减少无效留白。
- 提高列表密度。
- 保持触屏按钮可点击，不把关键按钮压得过小。
- 信息优先级明确，售卖页优先展示商品、价格、库存和购物车状态。

建议调整：

- 顶部导航高度压缩。
- 页面主标题控制在约 24px 到 32px。
- 卡片内边距和外边距整体收紧。
- 商品管理列表更接近表格/清单。
- 售卖页商品行高度稳定。
- 设置页表单分组压缩，但保留可读性。
- 操作按钮使用图标加短文字或纯图标加 tooltip，避免长文案撑大布局。

### 4.6 优惠 SPU 下拉

设置页加购优惠配置从手填改为下拉选择。

数据来源：

- 从商品管理中读取所有商品的 SPU。
- 去除空值。
- 去重。
- 排序展示。

下拉项建议展示：

```text
SPU 名称（n 个商品）
```

如果当前配置中的 SPU 已不存在：

- 仍显示当前值。
- 增加提示：“当前商品库未找到该 SPU，请确认是否已停用或删除相关商品。”
- 不自动清空配置，避免导入旧数据后规则丢失。

V1.1a 不改变加购优惠计算逻辑。

## 5. V1.1b 商品编码设计

### 5.1 字段定义

商品模型新增：

- `spuCode`：SPU 编码，用于标识商品款/产品主体。
- `skuCode`：SKU 编码，用于标识具体规格/可售单元。
- `productCode`：完整商品编码，用于展示、搜索、对账和现场识别。

保留现有：

- `spu`：SPU / 商品分组，用于业务分组、筛选、加购优惠、满赠按 SPU 配置。

不要把 `spu` 和 `spuCode` 混为同一个字段。

### 5.2 编码示例

用户示例：

```text
SPU 编码：CLTH-24001
SKU 编码：BLK-M
完整商品编码：CLTH-24001-BLK-M
```

完整商品编码由系统生成：

```text
完整商品编码 = SPU 编码 + "-" + SKU 编码
```

如果只填 SPU 编码而 SKU 编码为空，系统可预览 SPU 编码本身。但 V1.1b 起，新增或编辑保存商品时要求 SPU 编码和 SKU 编码均必填，以避免可售商品、非售卖库存品和赠品 SKU 无法区分。

### 5.3 SKU 编码重复前缀校验

如果用户录入：

```text
SPU 编码：CLTH-24001
SKU 编码：CLTH-24001-BLK-M
```

保存时阻止，并提示：

```text
SPU 编码与 SKU 编码存在重复内容。SKU 编码只需填写规格/变体部分，例如：BLK-M。
完整商品编码将由系统自动生成：CLTH-24001-BLK-M。
```

规则：

- 如果 `skuCode` 以 `spuCode` 开头，则视为 SPU/SKU 编码有内容重复。
- 比较前应去除首尾空格。
- 是否大小写敏感可按实现简化，但提示文案必须明确重复问题。

### 5.4 唯一性

完整商品编码不允许重复。

保存商品时应检查：

- 新增商品：不能与任何现有商品完整商品编码重复。
- 编辑商品：不能与其他商品完整商品编码重复。

如果历史数据没有编码：

- 兼容展示为空或显示“未设置编码”。
- 允许用户后续编辑补齐。
- 不在迁移时自动生成含义不明确的编码。

### 5.5 UI 展示

商品添加/编辑表单新增：

- SPU / 商品分组。
- SPU 编码。
- SKU 编码。
- 完整商品编码预览，只读。

商品管理列表展示：

- 商品图。
- 商品名称。
- 完整商品编码。
- SPU / 商品分组。
- 售价。
- 库存。
- 状态。
- 操作。

售卖页紧凑列表展示：

- 商品缩略图。
- 商品名称。
- 完整商品编码。
- SPU / 商品分组。
- 售价。
- 库存。
- 加购按钮。

购物车展示：

- 缩略图。
- 商品名称。
- 完整商品编码。
- 数量调节。
- 小计。

设置页指定 SKU 下拉展示：

```text
完整商品编码 - 商品名称
```

## 6. V1.1b 赠品配置设计

### 6.1 配置模型

赠品档位中每条赠品支持两种目标类型：

- 指定 SKU。
- 指定 SPU。

指定 SKU：

- 选择具体商品。
- 订单保存时直接扣减该商品库存。

指定 SPU：

- 选择某个 SPU / 商品分组。
- 购物车显示赠品权益。
- 下单/确认收款前必须选择实际发出的 SKU。
- 订单保存时按实际选择的 SKU 扣减库存。

### 6.2 指定 SPU 赠品的下单流程

示例配置：

```text
满 68 元：赠品A SPU x2
```

当购物车达到门槛：

1. 购物车显示“已达到 68 档：赠品A x2”。
2. 点击下单进入结账。
3. 如果该档包含指定 SPU 赠品，结账页显示赠品选择区。
4. 操作人从该 SPU 下可作为赠品且有库存的 SKU 中选择实际发出的赠品。
5. 选择数量必须满足赠品数量。
6. 未选择足够赠品 SKU 时，禁止确认收款。
7. 确认收款后，订单保存实际赠出的 SKU，并扣减对应库存。

### 6.3 赠品库存规则

指定 SKU：

- 检查该 SKU 库存是否满足赠品数量。

指定 SPU：

- 检查该 SPU 下可作为赠品的 SKU 总库存是否满足赠品数量。
- 结账选择时不能选择超过单个 SKU 当前库存的数量。
- 如果总库存不足，禁止确认收款并提示缺货。

### 6.4 订单记录

订单应保存：

- 触发的赠品档位。
- 原始赠品配置快照。
- 实际赠出的商品 SKU 明细。

订单明细中赠品行应使用实际商品 ID，而不是只保存 SPU。

## 7. 数据迁移与备份兼容

### 7.1 商品数据兼容

已有商品缺少 `spuCode`、`skuCode`、`productCode` 时：

- 应正常加载。
- 商品管理中显示为空或“未设置编码”。
- 用户编辑保存后补齐。

迁移不应自动用内部 ID 生成商品编码，因为内部 ID 不具备业务含义。

### 7.2 促销配置兼容

已有 `addonDiscount.discountSpu` 继续有效。

V1.1a 只是把设置页输入控件从手填改为下拉，不改变字段含义。

### 7.3 赠品配置兼容

已有赠品配置是指定商品 ID，应迁移或解释为：

```text
targetType: "sku"
productId: 原 productId
```

V1.1b 应保留旧备份导入能力。

### 7.4 备份

备份导出需要包含新增字段：

- `spuCode`
- `skuCode`
- `productCode`

导入时：

- 允许缺少这些字段的旧备份。
- 对新备份校验完整商品编码唯一性。
- 对赠品配置同时兼容旧结构和新结构。

- 可切换到图片网格，再切回紧凑列表。

- 加购按钮在两种视图下均能正确加入购物车。

## 9. 风险与控制

### 9.1 UI 密度风险

压缩界面不能牺牲触屏可用性。按钮和数量调节控件需要保持足够点击面积。

控制方式：

- 售卖页关键按钮不低于合理触屏尺寸。
- 文本过长时截断或换行，不能互相覆盖。
- 在桌面和 iPad 尺寸下检查布局。

### 9.2 赠品 SPU 库存风险

指定 SPU 赠品如果自动扣库存，容易和实际发放不一致。

控制方式：

- 已确认采用手动选择实际赠出 SKU。
- 未选择足够赠品 SKU 时不允许确认收款。

### 9.3 数据模型风险

商品编码和赠品配置会影响备份、订单、库存和测试。

控制方式：

- 先完成 V1.1a。
- V1.1b 单独实施和回归。
- 保持旧数据兼容。

## 10. 实施建议

推荐执行顺序：

1. V1.1a 设计确认。
2. 编写 V1.1a 实现计划。
3. 使用测试驱动方式改造售卖页、购物车和设置页。
4. 运行自动测试和构建。
5. 进行本地浏览器手动验收。
6. 再启动 V1.1b 商品编码与赠品规则实现计划。

V1.1a 完成前不建议开始 V1.1b，以避免 UI 回归和数据模型回归同时出现。

## 11. 当前确认决策

- 采用分段方案 A。
- 商品技术内部 ID 继续保留，不作为主要现场识别码。
- 新增用户可读商品编码体系。
- SPU 编码标识商品款/产品主体。
- SKU 编码标识具体规格/可售单元。
- 完整商品编码由系统自动生成。
- SKU 编码重复包含 SPU 编码时保存报错。
- 报错信息必须说明 SPU/SKU 编码存在重复内容。
- 赠品指定 SPU 时，确认收款前必须手动选择实际赠出的 SKU。
- 不做自动扣 SPU 下某个 SKU 的库存。
