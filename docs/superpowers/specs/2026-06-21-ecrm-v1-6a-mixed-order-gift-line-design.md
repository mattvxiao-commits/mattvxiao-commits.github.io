# ECRM V1.6a 混合订单、赠送明细与真实销售口径设计方案

## 1. 背景

V1.5.4 已支持现场开单、手动收款、订单详情、退款、作废、库存扣减、JSON 备份、订单 Excel 导出、固定仪表盘和现场模式。经过两天线下摆摊实操后，发现当前系统仍缺少一类高频现场能力：**在同一笔订单内同时记录真实销售商品和非销售赠送商品**。

现场实际发生了三类情况：

1. 用户正常消费后，参与店主设置的运营活动，例如关注小红书账号、关注其他社媒平台、加入社群或完成现场互动，额外获得一个小赠品。
2. 好友来摊位捧场，摊主直接免费赠送一批商品。
3. 因用户情况、补偿、临时判断等原因，即使未达到满赠档位，也额外赠送某个原本不可售、但可作为赠品的商品。

V1.5.4 中，操作者只能用以下绕路方式处理：

- 把赠礼拆成独立订单。
- 收款方式选择“其他”。
- 再进入订单详情执行退款。
- 或临时把不可售赠品改成可售、售价改为 0，再加入购物车。

这些 workaround 可以保住库存流水，但会带来明显问题：

- 操作步骤多，现场容易漏操作或误操作。
- “其他收款 + 退款”无法表达真实业务原因。
- 免费赠送和运营赠礼会污染销售额、客单价、热销排行、毛利、低毛利 SKU 等仪表盘指标。
- 不可售赠品为了出库被临时改成可售，会破坏商品主数据。
- 前两天已经产生的历史数据需要一种统计口径修正能力，否则仪表盘无法精准复盘。

## 2. 目标

V1.6a 的目标是建立明细级的销售 / 非销售出库模型，让 ECRM 能真实记录现场事实，并让仪表盘默认展示可信的真实销售数据。

核心目标：

1. 一笔订单内可以同时包含正常销售商品和赠送 / 非销售出库商品。
2. 非销售明细自动 0 元，不计入应收金额。
3. 非销售明细扣减库存，保留成本快照，计入对应赠送 / 出库成本。
4. 运营赠礼限定只能选择“可作为赠品”的商品，并记录当前运营活动名称。
5. 人工赠送、其他非销售出库允许选择全部商品，但必须填写原因 / 备注。
6. 订单详情清晰展示每一行明细的统计类型和原因。
7. 历史订单支持整单和明细级统计口径修正。
8. 仪表盘支持统计口径筛选，默认只看正常销售。
9. Excel 导出增加明细类型、非销售原因和修正信息，便于离线复盘。
10. JSON 备份兼容新字段，旧备份导入后默认按正常销售处理。

## 3. 非目标

V1.6a 不处理以下内容：

- 不接入小红书、微博、社群等外部平台的关注或互动状态自动验证。
- 不接入微信、支付宝支付 API 或自动查账。
- 不重构完整商品级退款、退货入库、换货、补差价、退差价。
- 不实现完整库存盘点、入库、损耗管理。
- 不实现云同步、多设备自动合并。
- 不实现复杂活动引擎或拖拽式仪表盘。
- 不删除或重写既有历史订单、退款和库存流水。

## 4. 核心概念

### 4.1 订单整体性质

订单整体性质可由订单明细派生，不建议第一版让用户直接编辑为唯一来源。

派生类型：

- `sale`：整单只有正常销售明细。
- `mixed`：同一订单同时包含正常销售明细和非销售明细。
- `non_sales`：整单只有非销售明细。

页面展示文案：

- `sale`：正常销售。
- `mixed`：销售 + 赠送。
- `non_sales`：非销售出库。

### 4.2 明细收入类型

订单明细新增收入类型：

- `sale`：正常销售。
- `non_sales`：非销售出库。

`sale` 明细：

- 计入应收金额。
- 计入销售额、客单价、销售排行和销售毛利。
- 扣库存。
- 使用下单时售价或促销计算价。

`non_sales` 明细：

- 单价自动为 0。
- 不计入应收金额、销售额、客单价、正常销售排行和正常销售毛利。
- 扣库存。
- 保留成本快照。
- 成本计入对应非销售出库成本。

### 4.3 非销售原因

非销售明细需要记录原因：

- `tier_gift`：满赠赠品，系统自动产生。
- `campaign_gift`：运营赠礼，操作员基于当前运营活动手动添加。
- `manual_gift`：人工赠送。
- `other_non_sales`：其他非销售出库。

用户可手动选择的类型：

- 正常销售。
- 运营赠礼。
- 人工赠送。
- 其他非销售出库。

系统自动产生的类型：

- 满赠赠品。

## 5. 明细类型规则

| 明细类型 | 是否计收入 | 是否扣库存 | 商品范围 | 是否强制备注 | 成本归属 | 经营归属 |
|---|---:|---:|---|---:|---|---|
| 正常销售 | 是 | 是 | 可售商品 | 否 | 销售成本 | 销售 |
| 加购优惠销售 | 是 | 是 | 促销配置 SPU | 否 | 销售成本 | 销售 + 优惠让利 |
| 满赠赠品 | 否 | 是 | 满赠配置 | 否 | 满赠成本 | 运营活动成本 |
| 运营赠礼 | 否 | 是 | 仅可作为赠品商品 | 否 | 运营赠礼成本 | 运营活动成本 |
| 人工赠送 | 否 | 是 | 全部商品 | 是 | 人工赠送成本 | 非经营赠送 |
| 其他非销售出库 | 否 | 是 | 全部商品 | 是 | 其他非销售出库成本 | 其他非销售出库 |

说明：

- 加购优惠销售仍属于销售明细，商品成本归销售成本；原价与成交价之间的差额单独记为优惠让利，不归为商品成本。
- 满赠赠品由既有满赠规则自动生成，应纳入非销售明细模型。
- 运营赠礼属于可配置运营活动，不允许选择非赠品商品。
- 人工赠送和其他非销售出库属于特殊情况，允许选择全部商品，但必须记录原因。

## 6. 成本与利润口径

V1.6a 使用经营管理口径，不追求会计报表级分类。目标是让摊主能快速判断：卖货本身是否赚钱、运营活动花了多少成本、非经营赠送和异常出库是否影响复盘。

### 6.1 收入

销售收入只来自销售明细：

```text
销售收入 = 正常销售实际收入 + 加购优惠销售实际收入
```

不计入销售收入：

- 满赠赠品。
- 运营赠礼。
- 人工赠送。
- 其他非销售出库。

### 6.2 优惠让利

加购优惠销售仍是销售明细。它的商品成本归销售成本，原价与实际成交价之间的差额归为优惠让利。

```text
优惠让利 = 加购优惠商品原价金额 - 加购优惠实际成交金额
```

示例：

```text
商品原价 5 元，加购价 3 元，成本 1 元
销售收入 = 3 元
销售成本 = 1 元
优惠让利 = 2 元
销售毛利 = 2 元
```

优惠让利用于评估活动牺牲了多少收入，不作为商品成本。

### 6.3 销售成本

销售成本只包含有收入的销售明细成本：

```text
销售成本 = 正常销售商品成本 + 加购优惠销售商品成本
```

销售成本不包含：

- 满赠成本。
- 运营赠礼成本。
- 人工赠送成本。
- 其他非销售出库成本。

### 6.4 基础销售毛利

基础销售毛利用于查看卖货本身是否赚钱：

```text
基础销售毛利 = 销售收入 - 销售成本
基础销售毛利率 = 基础销售毛利 / 销售收入
```

### 6.5 运营活动成本

运营活动成本只包含明确由经营活动规则产生的非销售成本：

```text
运营活动成本 = 满赠成本 + 运营赠礼成本
```

其中：

- 满赠成本：因满额活动产生的赠品成本。
- 运营赠礼成本：因配置的运营活动产生的赠礼成本，例如社媒关注、社群加入、现场互动等。

人工赠送和其他非销售出库不纳入运营活动成本，因为它们通常是人情、补偿、异常或记录修正，不应混入活动效果判断。

### 6.6 活动后毛利

活动后毛利用于查看销售收入扣除商品成本和运营活动成本后的经营结果：

```text
活动后毛利 = 销售收入 - 销售成本 - 运营活动成本
活动后毛利率 = 活动后毛利 / 销售收入
```

这是 V1.6a 仪表盘应优先提供的高价值指标之一。

### 6.7 非经营出库成本

非经营出库成本单独展示：

```text
非经营出库成本 = 人工赠送成本 + 其他非销售出库成本
```

它用于解释好友赠送、用户补偿、样品、异常处理、记录修正等对库存和成本的影响。默认不扣减活动后毛利，但可在“全部活动”或导出中单独查看。

### 6.8 全出库成本

全出库成本用于查看当天所有库存成本消耗：

```text
全出库成本 = 销售成本 + 运营活动成本 + 非经营出库成本
```

这个指标适合盘点和库存复盘，不适合作为默认销售毛利口径。

## 7. 购物车设计

### 7.1 正常销售默认行为

当前售卖页点击商品右侧 `+` 时，仍默认添加为正常销售明细。

正常销售明细继续参与：

- 加购优惠计算。
- 满赠档位判断。
- 应收金额计算。
- 库存扣减。

### 7.2 已有购物车行切换明细类型

购物车每一行增加“明细类型”控制。

可选项：

- 正常销售。
- 运营赠礼。
- 人工赠送。
- 其他非销售出库。

切换规则：

- 如果商品不是“可作为赠品”，不能切换为运营赠礼。
- 切换为运营赠礼、人工赠送、其他非销售出库后，单价自动为 0。
- 切换回正常销售后，恢复当前商品售价，并重新参与促销计算。
- 切换为人工赠送或其他非销售出库时，必须填写原因 / 备注后才能下单。

### 7.3 快捷添加入口

购物车或售卖页增加三个快捷入口：

- `+ 运营赠礼`
- `+ 人工赠送`
- `+ 其他出库`

点击后打开商品选择弹窗。

选择范围：

| 入口 | 默认范围 | 是否可显示全部商品 | 说明 |
|---|---|---:|---|
| 运营赠礼 | 可作为赠品商品 | 否 | 运营活动有明确限定范围 |
| 人工赠送 | 可作为赠品商品 | 是 | 人情、补偿、临时赠送 |
| 其他出库 | 可作为赠品商品 | 是 | 样品、异常处理、记录修正等 |

人工赠送和其他出库选择非赠品商品时，应提示：

```text
当前商品不是赠品商品，将按 0 元非销售出库并扣减库存。
```

### 7.4 原因 / 备注规则

运营赠礼：

- 不强制备注。
- 由系统记录当前选择的运营活动名称。
- 运营活动名称在设置页配置，示例：
  - 关注小红书赠礼。
  - 关注微博赠礼。
  - 加入社群赠礼。
  - 现场互动赠礼。
  - 其他运营活动。
- V1.6a 第一版至少支持配置活动名称和默认赠礼 SKU。

人工赠送：

- 必须选择原因。
- 快捷原因：
  - 好友赠送。
  - 用户补偿。
  - 摊主临时赠送。
  - 其他。
- 选择“其他”时，需要填写补充备注。

其他非销售出库：

- 必须选择原因。
- 快捷原因：
  - 样品。
  - 异常处理。
  - 记录修正。
  - 其他。
- 选择“其他”时，需要填写补充备注。

## 8. 订单金额与收款规则

订单应收金额只由 `sale` 明细决定。

示例：

```text
商品 A    正常销售    30 元
商品 B    正常销售    15 元
赠品 C    运营赠礼     0 元
商品 D    人工赠送     0 元
```

应收金额：

```text
45 元
```

支付方式仍只用于有收款事实的订单。

纯非销售出库订单：

- 应收金额为 0。
- 不应要求选择微信 / 支付宝 / 现金 / 其他收款。
- 可保存为非销售出库记录。
- 订单状态可使用已完成，或新增非销售完成状态，具体在实施计划中根据现有订单状态选择最小改动方案。

混合订单：

- 需要对销售部分正常收款。
- 非销售明细不增加应收金额。

## 9. 运营赠礼配置

V1.6a 不把运营赠礼写死为某个平台或某个活动。设置页需要新增轻量配置，让不同摆摊场次可以调整活动名称和默认赠礼。

建议配置项：

- 是否启用运营赠礼。
- 活动名称。
- 默认赠礼 SKU。
- 是否要求本单存在正常消费。

V1.6a 默认：

- 启用后，购物车显示 `+ 运营赠礼`。
- 活动名称必填。
- 默认赠礼 SKU 必须是可作为赠品的商品。
- 运营赠礼必须依附至少一条正常销售明细。

活动名称示例：

```text
关注小红书赠礼
关注微博赠礼
加入社群赠礼
现场互动赠礼
转发截图赠礼
```

订单明细保存时需要记录活动名称快照，避免后续设置修改影响历史订单。

## 10. 订单详情设计

订单详情需要展示：

- 订单整体性质：正常销售、销售 + 赠送、非销售出库。
- 应收金额。
- 销售金额。
- 非销售出库件数。
- 非销售出库成本。
- 每行明细类型。
- 每行非销售原因 / 备注。

示例：

```text
订单性质：销售 + 赠送
销售金额：45 元
非销售出库：2 件
非销售成本：3.40 元
```

明细：

```text
商品 A    正常销售      30 元
商品 B    正常销售      15 元
赠品 C    运营赠礼       0 元
商品 D    人工赠送       0 元    原因：好友赠送
```

## 11. 历史订单修正

### 11.1 修正目标

前两天摆摊已经产生了“其他收款 + 退款”的赠礼订单，也产生了临时把赠品改为可售、售价改为 0 的订单。V1.6a 需要提供统计口径修正能力，让历史数据可以重新进入正确仪表盘口径。

### 11.2 修正原则

- 不删除原订单。
- 不删除退款记录。
- 不删除或重写库存流水。
- 不直接伪造原始支付事实。
- 通过新增或更新统计口径字段影响仪表盘和导出。
- 订单详情中保留原始记录和修正后口径。

### 11.3 整单修正

适用于整单都是某类非销售出库的情况。

可将整单明细批量修正为：

- 运营赠礼。
- 人工赠送。
- 其他非销售出库。

整单修正时：

- 所有明细销售单价在统计口径中视为 0。
- 原订单金额和退款记录仍保留。
- 仪表盘按修正后的明细类型计算。
- 人工赠送和其他非销售出库仍需填写原因 / 备注。

### 11.4 明细修正

适用于一单内只有部分商品需要修正。

可将某一行明细修正为：

- 正常销售。
- 运营赠礼。
- 人工赠送。
- 其他非销售出库。

限制：

- 修正为运营赠礼时，目标商品必须是可作为赠品商品。
- 修正为人工赠送或其他非销售出库时，必须填写原因 / 备注。
- 修正只影响统计口径，不倒推修改库存流水。

## 12. 仪表盘口径

仪表盘新增统计口径筛选：

- 正常销售。
- 全部活动。
- 运营赠礼。
- 人工赠送。
- 其他非销售出库。

默认口径：

- 正常销售。

### 12.1 正常销售

只统计 `sale` 明细。

用于：

- 销售额。
- 订单数。
- 客单价。
- 成本。
- 毛利。
- 毛利率。
- SKU/SPU 销售排行。
- SKU/SPU 毛利排行。
- 低毛利 SKU。

非销售明细不进入这些指标。

### 12.2 全部活动

展示销售与非销售出库的完整现场活动。

建议展示：

- 销售额：只来自销售明细。
- 优惠让利。
- 销售成本。
- 基础销售毛利。
- 满赠成本。
- 运营赠礼成本。

## 16. 数据模型建议

订单明细建议新增字段：

```ts
type OrderLineRevenueType = "sale" | "non_sales";

type NonSalesReason =
  | "tier_gift"
  | "campaign_gift"
  | "manual_gift"
  | "other_non_sales";

type OrderItem = {
  revenueType?: OrderLineRevenueType;
  nonSalesReason?: NonSalesReason;
  nonSalesNote?: string;
  campaignNameSnapshot?: string;
  discountGiveawayAmount?: number;
  statisticalUnitPrice?: number;
  statisticalSubtotal?: number;
  originalRevenueType?: OrderLineRevenueType;
  adjustedAt?: string;
};
```

说明：

- 旧数据字段可选，读取时默认 `revenueType = sale`。
- `statisticalUnitPrice` 和 `statisticalSubtotal` 用于统计口径，不一定改写原始 `unitPrice`。
- 对新建非销售明细，原始单价可以直接保存为 0。
- 对历史修正明细，保留原始金额，同时统计金额按 0 处理。
- `campaignNameSnapshot` 用于保存下单时运营活动名称，避免设置变化影响历史订单。
- `discountGiveawayAmount` 用于保存加购优惠等销售明细的优惠让利金额。
- `originalRevenueType` 和 `adjustedAt` 用于识别是否经过统计口径修正。

实施计划中需要根据现有 `OrderItem` 类型、Dexie schema 和导入导出代码进一步收敛字段命名。

## 17. 促销规则影响

### 17.1 加购优惠

加购优惠只对正常销售明细生效。

加购优惠销售：

- 仍属于销售明细。
- 商品成本计入销售成本。
- 实际成交金额计入销售收入。
- 原价与成交价差额计入优惠让利。

非销售明细：

- 不触发加购优惠。
- 不占用加购优惠名额。
- 不参与“任意消费”金额计算。

### 17.2 满赠档位

满赠档位只由正常销售金额触发。

运营赠礼、人工赠送、其他非销售出库：

- 不提高满赠档位。
- 不参与满赠金额门槛。

系统自动生成的满赠赠品：

- 作为 `non_sales / tier_gift` 明细。
- 扣库存。
- 计入满赠成本。

### 17.3 运营赠礼

运营赠礼的业务前提是用户有任意消费并参与运营活动。

第一版建议：

- 操作员手动添加运营赠礼。
- 如果购物车没有正常销售明细，点击添加运营赠礼时提示：

```text
运营赠礼需要本单存在正常消费商品。
```

纯运营赠礼订单不允许保存。运营赠礼定义为“任意消费 + 参与运营活动”，必须依附至少一条正常销售明细。

## 18. 退款与作废影响

非销售明细不产生可退款金额。

混合订单最大可退款金额只由销售明细决定。

订单作废：

- 作废仍应回滚全部明细库存，包括销售明细和非销售明细。
- 作废后仪表盘不再统计销售与非销售出库。

人工退款：

- 退款仍记录为订单级。
- 仪表盘正常销售口径应在既有退款逻辑基础上计算。
- V1.6a 不做商品级退款拆分。

## 19. UI/UE 验收要求

### 19.1 购物车

- 纯非销售出库订单不应要求扫码收款。

- 每行类型清晰。

## 20. 风险与控制

### 风险 1：字段模型影响订单、仪表盘、导出、备份

控制：

- 先在领域层建立统一归一化函数。
- 旧订单明细读取时默认视为正常销售。
- 仪表盘、导出、订单详情都使用同一归一化口径。

### 风险 2：购物车促销计算复杂化

控制：

- 非销售明细不参与促销计算。
- 促销引擎只读取正常销售明细。
- 满赠自动生成的赠品统一为非销售明细。

### 风险 3：历史修正破坏原始事实

控制：

- 不删除退款记录。
- 不删除库存流水。
- 不改写原始支付事实。
- 修正只影响统计口径。

### 风险 4：人工赠送允许全部商品导致误操作

控制：

- 默认只显示赠品商品。
- 显示全部商品需要明确切换。
- 非赠品商品被选择时给出提示。
- 人工赠送和其他非销售出库必须填写原因。

## 22. 验收场景

### 场景 1：消费 + 运营赠礼

1. 加入正常销售商品 1 个。
2. 点击 `+ 运营赠礼`。
3. 选择可作为赠品商品。
4. 购物车应收只包含销售商品。
5. 下单后库存同时扣减销售商品和赠礼。
6. 仪表盘正常销售口径只统计销售商品。
7. 仪表盘运营赠礼口径统计赠礼数量和成本。

### 场景 2：好友人工赠送

1. 点击 `+ 人工赠送`。
2. 切换显示全部商品。
3. 选择普通可售商品。
4. 选择原因“好友赠送”。
5. 保存为纯非销售出库订单。
6. 不要求收款。
7. 库存扣减。
8. 仪表盘正常销售口径不统计该订单。
9. 仪表盘人工赠送口径统计数量和成本。

### 场景 3：其他非销售出库

1. 点击 `+ 其他出库`。
2. 选择商品。
3. 选择原因“异常处理”。
4. 下单后扣库存。
5. 仪表盘其他非销售出库口径可查看。

### 场景 4：历史订单整单修正

1. 打开一笔旧的“其他收款 + 已退款”赠礼订单。
2. 选择整单修正为人工赠送。
3. 填写原因。
4. 原始支付和退款记录仍显示。
5. 仪表盘正常销售口径排除该订单。
6. 仪表盘人工赠送口径统计该订单成本。

### 场景 5：历史订单明细修正

1. 打开一笔包含 0 元商品的旧订单。
2. 将 0 元商品明细修正为人工赠送。
3. 订单其他销售明细仍计入正常销售。
4. 仪表盘正常销售口径不再出现该商品的低毛利异常。

## 23. V1.6a 默认实施决策

当前已确认并纳入 V1.6a：

- 运营赠礼只能选择可作为赠品商品。
- 人工赠送和其他非销售出库可选择全部商品。
- 人工赠送和其他非销售出库必须填写原因 / 备注。
- 仪表盘需要支持正常销售、全部活动、运营赠礼、人工赠送、其他非销售出库口径。
- 历史订单需要支持整单和明细级统计口径修正。

实施默认决策：

1. 纯运营赠礼订单不允许保存。
   - 运营赠礼定义为“任意消费 + 参与运营活动”，必须依附至少一条正常销售明细。
2. V1.6a 支持设置页配置默认运营赠礼商品。
   - 支持选择默认赠礼 SKU。
   - 如果后续实现成本可控，再扩展为默认赠礼 SPU。
   - 购物车 `+ 运营赠礼` 优先使用默认赠礼，减少现场每单选择。
3. 人工赠送 / 其他非销售出库不允许数量大于库存。
   - 保持与销售扣库存一致。
   - 库存不足时必须提示并阻止保存。
4. 历史修正允许把非销售明细改回正常销售。
   - 需要保留修正记录。
   - 改回正常销售后，统计口径重新计入正常销售。
   - 原始退款、库存流水、支付记录仍不删除。
