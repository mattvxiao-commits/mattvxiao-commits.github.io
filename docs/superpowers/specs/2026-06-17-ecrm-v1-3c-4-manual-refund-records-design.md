# ECRM V1.3c-4 人工退款记录方案

## 1. 背景

V1.3b 已经实现已支付订单作废和库存回滚。V1.3c-1 至 V1.3c-3 继续补齐了作废原因、订单详情库存摘要，以及订单记录列表的作废标识。

现在还缺少一类现场高频售后记录：客户已经通过微信、支付宝、现金或其他方式完成线下退款，但系统里无法记录“退了多少钱、什么时候退、为什么退、是否已处理”。这会影响活动结束后的对账和 V1.4 仪表盘统计。

V1.3c-4 的目标是新增轻量人工退款记录能力。这里的“退款”只表示操作员已在线下完成退款并在系统内登记，不调用微信或支付宝接口。

## 2. 目标

本小版本交付以下能力：

- 已支付订单可以记录人工退款。
- 已作废订单也可以记录人工退款，用于标记作废后是否已向客户退回款项。
- 待支付订单不能记录退款。
- 一笔订单可以有多条人工退款记录。
- 退款记录包含金额、方式、原因、备注和记录时间。
- 单笔订单累计退款金额不能超过订单应收金额。
- 订单详情展示退款记录列表和累计退款金额。
- 订单记录列表可以显示“已退款”或“部分退款”标识。
- 备份导出和导入包含退款记录；旧版备份不包含退款记录时仍可导入。

## 3. 非目标

本小版本明确不做以下内容：

- 不接入微信退款 API。
- 不接入支付宝退款 API。
- 不生成自动退款流水。
- 不做商品级部分退款选择。
- 不做退款退货入库。
- 不改订单主状态，不新增 `refunded` 或 `partially_refunded`。
- 不新增订单调整表。
- 不做反向冲销记录。
- 不做 V1.4 仪表盘图表。

## 4. 业务规则

### 4.1 可记录退款的订单

允许记录退款：

- `paid`：已支付订单。
- `cancelled`：已作废订单。

不允许记录退款：

- `pending_payment`：待支付订单。
- 订单不存在。

原因：

- 已支付订单有真实收款事实，可以发生人工退款。
- 已作废订单代表订单已从有效销售中排除，但仍可能需要记录“是否已经把钱退给客户”。
- 待支付订单没有确认收款事实，登记退款会造成账目歧义。

### 4.2 退款金额

规则：

- 退款金额必须大于 0。
- 退款金额最多保留两位小数。
- 单笔订单累计退款金额不能超过订单 `payableAmount`。
- 可以一次记录全额退款，也可以多次记录部分金额退款。

示例：

- 订单应收 `100`，已有退款 `40`，本次最多可再记录 `60`。
- 订单应收 `100`，本次直接记录 `100` 后，后续不能继续记录退款。

### 4.3 退款方式

沿用现有支付方式枚举：

- `wechat`：微信。
- `alipay`：支付宝。
- `cash`：现金。
- `other`：其他。

记录退款时的默认值：

- 如果原订单有支付方式，默认使用原订单支付方式。
- 如果原订单没有支付方式，默认使用 `cash`。

### 4.4 退款原因

新增退款原因枚举：

- `customer_return`：客户退单。
- `overcharge`：多收款。
- `product_issue`：商品问题。
- `manual_adjustment`：人工调整。
- `other`：其他。

默认原因：

- `customer_return`。

备注：

- 备注可选。
- 保存时会去掉首尾空格。
- 空备注存为 `undefined`。

### 4.5 订单状态与统计口径

V1.3c-4 不改变订单主状态。

原因：

- 退款是售后记录，不代表订单从未发生。
- 同一订单可能多次退款，用单一订单状态无法表达退款金额、方式和备注。
- 作废与退款不是同一件事：作废影响订单有效性，退款记录表达现金流处理。

后续 V1.4 仪表盘建议口径：

- 原始销售额：所有已支付订单应收金额。
- 作废订单金额：作废订单应收金额。
- 人工退款金额：退款记录金额合计。
- 净销售额：需要避免作废和退款重复扣减。已作废订单的退款记录主要用于现金流追踪，不能再次从净销售额中重复扣除。

## 5. 数据设计

### 5.1 新增类型

新增 `OrderRefund`：

```ts
export type RefundReason =
  | "customer_return"
  | "overcharge"
  | "product_issue"
  | "manual_adjustment"
  | "other";

export type OrderRefund = {
  id: string;
  orderId: string;
  amount: number;
  method: PaymentMethod;
  reason: RefundReason;
  note?: string;
  createdAt: string;
};
```

### 5.2 Dexie 表

新增 `orderRefunds` 表。

索引：

- `id`
- `orderId`
- `createdAt`

数据库升级：

- 从 `version(1)` 升级到 `version(2)`。
- 保留现有表结构。
- 新增 `orderRefunds: "id, orderId, createdAt"`。

### 5.3 备份版本

备份版本从 `2` 升级到 `3`。

版本兼容：

- 版本 1：无图片、无退款记录，导入时 `images: []`、`orderRefunds: []`。
- 版本 2：有图片、无退款记录，导入时 `orderRefunds: []`。
- 版本 3：有图片、有退款记录。

导出：

- `data.orderRefunds` 必须包含所有退款记录。

导入：

- 校验每条退款记录字段。
- 校验 `amount > 0`。
- 校验 `method` 属于支付方式枚举。
- 校验 `reason` 属于退款原因枚举。
- 校验 `orderId` 是字符串。
- 校验 `createdAt` 是字符串。
- 暂不强制校验 `orderId` 一定存在于订单列表，避免旧备份或人工修复备份被过早拒绝；repository 运行时仍会校验真实订单。

## 6. Repository 设计

新增函数：

```ts
export type SaveOrderRefundInput = {
  orderId: string;
  amount: number;
  method: PaymentMethod;
  reason: RefundReason;
  note?: string;
};

export async function saveOrderRefund(input: SaveOrderRefundInput, now = new Date()): Promise<OrderRefund>;

export async function listOrderRefunds(orderId: string): Promise<OrderRefund[]>;

export async function listRefunds(): Promise<OrderRefund[]>;
```

`saveOrderRefund` 规则：

- 必须在事务内读取订单和当前订单退款记录。
- 订单不存在时拒绝。
- 订单状态为 `pending_payment` 时拒绝。
- 金额不是有限数字或小于等于 0 时拒绝。
- 金额超过剩余可退金额时拒绝。
- 保存前把金额四舍五入到两位小数。
- 保存前把备注 trim，空字符串存为 `undefined`。
- 记录 ID 使用现有 `makeId("refund")`。

错误文案必须是固定中文，避免向 UI 暴露底层异常：

- `订单不存在，无法记录退款。`
- `待支付订单不能记录退款。`
- `退款金额必须大于 0。`
- `退款金额不能超过订单剩余可退金额。`

## 7. 页面与交互

### 7.1 订单详情

订单详情新增退款能力。

展示内容：

- 售后记录区常驻条件调整为：订单已作废，或存在退款记录，或订单可记录退款。
- 展示累计退款金额。
- 展示剩余可退金额。
- 展示退款记录列表。
- 每条退款记录显示金额、方式、原因、时间和备注。

操作按钮：

- `记录退款`。

显示规则：

- `paid`：显示 `作废订单` 和 `记录退款`。
- `cancelled`：不显示 `作废订单`，显示 `记录退款`。
- `pending_payment`：不显示售后操作。
- 累计退款金额已达到订单应收金额时，不再显示 `记录退款`，或按钮置灰并提示“已退满订单金额”。V1.3c-4 推荐直接隐藏按钮，减少误操作。

### 7.2 记录退款弹窗

字段：

- 退款金额。
- 退款方式。
- 退款原因。
- 备注。

校验：

- 金额必填。
- 金额必须大于 0。
- 金额不能超过剩余可退金额。
- 校验错误在弹窗内显示中文红字。

保存后：

- 关闭弹窗。
- 刷新订单详情退款记录。
- 售卖页显示成功提示：`订单 订单号 已记录退款 ¥金额。`

失败时：

- 售卖页显示固定中文：`退款记录保存失败，请刷新后重试。`
- 不显示底层异常。

### 7.3 订单记录列表

订单列表标识扩展：

- 无退款：保持现状。
- 退款金额大于 0 且小于订单应收：显示 `部分退款`。
- 退款金额大于等于订单应收：显示 `已退款`。
- 作废订单仍显示 `已作废`、作废原因和 `有备注`。

V1.3c-4 为避免订单列表每行都额外查询退款记录，推荐先在售卖页加载订单时同时加载全部退款记录，并传入领域函数计算标识。

## 8. 测试策略

### 8.1 数据层测试

覆盖：

- 已支付订单记录退款成功。
- 已作废订单记录退款成功。
- 待支付订单拒绝记录退款。
- 订单不存在拒绝记录退款。
- 金额为 0、负数、非有限数字时拒绝。
- 累计退款金额超过订单应收时拒绝。
- 退款记录按 `createdAt` 升序查询。

### 8.2 备份测试

覆盖：

- 版本 3 导出包含 `orderRefunds`。
- 版本 3 导入恢复 `orderRefunds`。
- 版本 1/2 旧备份导入时 `orderRefunds` 默认为空。
- 非法退款金额导入失败。
- 非法退款方式导入失败。
- 非法退款原因导入失败。

### 8.3 组件测试

覆盖：

- 订单详情展示累计退款金额和退款记录。
- 已支付订单显示 `记录退款`。
- 已作废订单显示 `记录退款`。
- 待支付订单不显示 `记录退款`。
- 退款弹窗校验必填金额和超额金额。

### 8.4 页面测试

覆盖：

- 打开订单详情时加载退款记录。
- 成功记录退款后刷新订单详情。
- 保存失败时显示固定中文错误。
- 订单列表显示 `部分退款` 或 `已退款`。

## 9. 验收标准

- 已支付订单可以记录人工退款。
- 已作废订单可以记录人工退款。
- 待支付订单不能记录人工退款。
- 退款金额不能超过订单应收金额。
- 订单详情可看到退款记录和累计退款金额。
- 订单列表可识别部分退款和已退款订单。
- 导出/导入备份后退款记录不丢失。
- 旧备份仍可导入。
- 全量测试通过。
- 生产构建通过。

## 10. 推进建议

V1.3c-4 建议拆成三个提交：

1. 数据层：类型、Dexie 表、repository、数据层测试。
2. 备份层：备份版本 3、导入导出校验、备份测试。
3. UI 层：订单详情退款弹窗、售卖页加载与保存、订单列表退款标识、页面测试。

这样可以把数据库升级风险、备份兼容风险和 UI 风险拆开验证，避免一次提交里混合多个问题。
