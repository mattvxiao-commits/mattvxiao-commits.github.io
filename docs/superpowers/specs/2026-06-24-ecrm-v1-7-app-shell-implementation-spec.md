# ECRM V1.7 App Shell 中文实施规格

## 1. 目标

V1.7 的目标是完成 ECRM 全局 GUI 壳层改版：将当前顶部导航为主的结构，调整为更适合桌面端长期扩展的左侧窄导航结构，同时保留当前正式版已经完成的商品、售卖、订单、仪表盘、设置等业务内容与交互能力。

本规格用于约束正式开发阶段，避免把 `docs/prototypes/v1-7-app-shell/hybrid-desktop.html` 中的简化原型内容误当作正式功能标准。

核心原则：

- 原型决定壳层、导航、空间结构、页面承载方式。
- 正式 `src` 代码决定业务内容、字段、组件细节、数据逻辑和已有交互。
- V1.7 不应因为 UI 壳层调整而删除、弱化或回退 V1.6.x 已实现的正式功能。

## 2. 范围

本阶段处理：

- 桌面端 App Shell 基础结构。
- 左侧窄导航菜单。
- 现场模式状态入口。
- 商品、售卖、订单、数据、设置页面在新壳层中的承载。
- 页面顶部筛选区、主操作区和内容区的密度统一。
- 现有正式组件在新壳层下的布局适配。

本阶段不处理：

- 不重写商品、订单、促销、库存、退款、仪表盘统计等业务逻辑。
- 不删除正式版已有字段和模块。
- 不把原型 HTML 的简化 DOM 直接迁移到正式 React 页面。
- 不做复杂飞书式仪表盘配置系统。
- 不做云同步、多设备实时同步、账号系统。

## 3. 原型参考边界

`hybrid-desktop.html` 只作为 V1.7 GUI 壳层参考，参考内容包括：

- 左侧窄导航的方向。
- 菜单顺序和名称。
- 现场模式状态入口的表现方式。
- 页面工作区宽度、边距和整体密度。
- 购物车抽屉、收款弹层、订单详情弹层在新壳层下的层级关系。
- 订单页独立成页后的双列列表方向。

不应直接参考或覆盖的内容：

- 原型中的购物车明细字段数量。
- 原型中的收款页字段完整度。
- 原型中的订单详情字段完整度。
- 原型中的仪表盘统计计算。
- 原型中的设置页业务逻辑。

这些内容必须以正式 `src` 目录中的现有组件和页面为基准。

## 4. 必须保留的原型决策

### 4.1 左侧窄导航

桌面端采用左侧固定窄导航，不再使用顶部主导航作为桌面端主菜单。

菜单顺序：

```text
现场状态 / 商品 / 售卖 / 订单 / 数据 / 设置
```

其中“现场状态”在 GUI 原型中表现为 `未锁定 / 现场模式`，不作为普通页面入口。

普通菜单显示规则：

- 图标在上。
- 两字中文在下。
- 菜单包括：商品、售卖、订单、数据、设置。
- 选中态使用统一的浅绿色强调。

### 4.2 现场模式状态入口

现场模式入口与普通菜单按钮做视觉区分。

默认未开启现场模式：

- 文案显示：`未锁定`。
- 图标为打开的锁。
- 图标位于文字后方，并与文字约 35% 重叠。
- 文字需要有接近导航底板色的描边或等效处理，确保识别清楚。

开启现场模式：

- 文案显示：`现场模式`。
- 图标为锁住状态。
- 图标压在文字上方，表达锁定状态。

交互：

- 点击该入口不切换页面。
- 点击后仅显示 3 秒 Tips。
- 未开启时提示：`现场模式未开启`。
- 已开启时提示：`现场模式已启动，页面已锁定`。

### 4.3 商品系列筛选

商品新增 / 编辑弹窗必须保留 `系列（筛选）` 字段。

字段规则：

- 非必填。
- 文本输入。
- 默认为空。
- 用于支持同人摊主按 IP、系列、作品等维度快速筛选商品。

售卖页配套规则：

- 当可售商品中不存在系列，或只有 1 个不同系列时，不显示系列筛选。
- 当可售商品中存在 2 个及以上不同非空系列时，在 SPU 筛选区上方显示系列筛选区。
- 系列筛选和 SPU 筛选联动：切换系列后，SPU 筛选应基于当前系列内商品重新计算。

### 4.4 商品编码

商品新增 / 编辑弹窗必须保留 SPU 编码、SKU 编码、完整商品编码。

排列：

- 第一行：商品名称、系列（筛选）、SPU 名称。
- 第二行：SPU 编码、SKU 编码、完整商品编码。
- 第三行：成本价、售价、库存数量。
- 第四行：可售卖、可赠品。

规则：

- 完整商品编码由系统自动组合。
- 组合方式：`SPU 编码 + "-" + SKU 编码`。
- 完整商品编码只读显示。
- SKU 编码只填写规格/变体部分，不应重复 SPU 编码内容。

### 4.5 独立订单页

V1.7 必须保留 GUI 原型中拆分出的独立订单页。

订单页要求：

- 订单记录不再只依附在售卖页中。
- 桌面端订单列表采用双列并排。
- 按订单生成顺序两两成组。
- 每组内部左旧右新。
- 最新一组显示在最上方。
- 示例：8 单时显示为 `007 008 / 005 006 / 003 004 / 001 002`。
- 单条订单高度应压缩，减少空间浪费。

订单单条标签：

- 支付方式使用标签展示。
- 不同支付方式采用不同配色。
- 微信、支付宝、现金、其他需要有明确区分。
- 订单状态、订单性质和支付方式标签应成组展示。
- 金额与标签之间需要有清晰间距。

### 4.6 设置页现场模式锁定范围

设置页现场模式模块必须保留 `锁定范围` 多选区。

要求：

- 位于现场模式设置模块内。
- 可表达锁定哪些页面或功能模块。
- 默认至少包含：商品、订单详情、数据、设置。
- 正式实现时，最终锁定范围应以该配置为准。

### 4.7 设置页现场模式 PIN

设置页现场模式模块需要保留两次 PIN 输入：

- 设置现场模式 PIN。
- 确认现场模式 PIN。

按钮：

- 更新 PIN。
- 立即重新锁定。
- 关闭现场模式。

状态展示：

- 使用标签展示当前状态。
- 不使用只读输入框展示状态。

## 5. 正式组件内容基准

以下正式组件或页面内容必须以 `src` 现有实现为基准，不得因原型简化而删减：

- `src/components/CartPanel.tsx`
- `src/components/CheckoutPanel.tsx`
- `src/components/OrderDetailDialog.tsx`
- `src/pages/DashboardPage.tsx`
- `src/pages/SettingsPage.tsx`
- `src/pages/SalesPage.tsx`
- `src/pages/ProductsPage.tsx`

具体要求：

- 购物车必须保留正式版已有的商品排序、赠品库存提示、非销售出库备注、促销摘要、金额区、清空 / 暂存 / 去收款按钮。
- 收款页必须保留正式版已有的收款方式、二维码、赠品选择、库存校验、确认收款逻辑。
- 订单详情必须保留正式版已有的基本信息、售后记录、商品明细、成本毛利、库存摘要、完整库存流水、退款、作废、统计口径修正能力。
- 数据页必须保留正式版已有的时间周期筛选、统计口径筛选、经营概览、销售表现、成本毛利、活动成本、非销售出库、库存异常等内容。
- 设置页必须保留正式版已有的基础信息、收款码、促销配置、运营赠礼、备份恢复、订单 Excel 导出、系统信息等模块。

## 6. 实施步骤

### 6.1 V1.7a：App Shell 基础结构

目标：

- 建立正式 React App Shell。
- 桌面端使用左侧窄导航。
- 页面主区域适配新壳层。
- 保留当前路由和页面组件。

验收：

- 商品、售卖、订单、数据、设置均可通过左侧导航进入。
- 现场模式入口不进入页面，只显示状态 Tips。
- 现有页面内容没有被删除。

### 6.2 V1.7b：页面顶部与筛选区适配

目标：

- 商品、售卖、订单、数据、设置页面顶部结构适配新壳层。
- 减少标题、说明、按钮、筛选区的空间浪费。
- 保留正式版所有筛选和操作能力。

验收：

- 售卖页系列筛选和 SPU 筛选逻辑正常。
- 数据页周期筛选和统计口径筛选正常。
- 订单页筛选和订单记录入口正常。

### 6.3 V1.7c：购物车、收款、订单详情适配

目标：

- 购物车抽屉适配新壳层。
- 收款页适配新壳层。
- 订单详情弹窗适配新壳层。

验收：

- 购物车正式功能不回退。
- 收款正式功能不回退。
- 订单详情所有正式模块仍存在。
- 退款、作废、统计口径修正仍可用。

### 6.4 V1.7d：订单页与数据页适配

目标：

- 新增或完善独立订单页。
- 订单列表采用双列布局。
- 数据页在新壳层中保持正式版统计内容。

验收：

- 订单列表按生成顺序两两成组，组内左旧右新，最新一组在最上方。
- 支付方式标签颜色区分清楚。
- 数据页统计内容与 V1.6.x 正式版一致，不因 UI 改版缺失。

### 6.5 V1.7e：设置页和现场模式适配

目标：

- 设置页现场模式模块按新规格调整。
- 增加锁定范围多选区。
- 保留设置页正式版全部模块。

验收：

- 两次 PIN 输入校验正常。
- 更新 PIN、立即重新锁定、关闭现场模式可用。
- 锁定范围配置可保存并影响实际锁定范围。
- 基础信息、收款码、促销、运营赠礼、备份、订单 Excel、系统信息仍可用。

## 7. 测试要求

每个小版本至少运行：

```text
npm test -- --run
npm run build
```

涉及 UI 结构变化时，需要补充人工验收：

- 桌面宽屏。
- 窄桌面窗口。
- iPad 横屏。
- iPad 竖屏或移动端后续方案。

关键回归点：

- 商品新增、编辑、复制。
- 售卖开单、购物车、收款。
- 运营赠礼、人工赠送、其他出库。
- 订单详情、退款、作废、统计口径修正。
- 仪表盘筛选和各统计模块。
- 设置页导入导出、订单 Excel 导出、现场模式。

## 8. 风险控制

风险：

- 原型页面内容不完整，可能误导正式实现。
- 壳层改动影响所有页面布局。
- 购物车、收款、订单详情、仪表盘、设置页已有内容较多，容易在重排时遗漏。

控制方式：

- 正式开发时禁止直接复制原型 HTML 替代 React 组件。
- 每个页面改造前先阅读对应正式组件。
- 每个页面改造后做内容清单对照。
- 对正式功能使用现有测试兜底，必要时补充新测试。
- 每个小版本单独提交，避免大批量改动难以回滚。

## 9. 结论

V1.7 正式实现应采用“壳层替换、内容保留”的方式推进。

`hybrid-desktop.html` 只作为 GUI 壳层和布局密度参考；正式组件的内容、字段、按钮、业务逻辑和测试，以 `src` 现有实现为准。
