# ECRM V1.3c 订单售后规则方案

## 1. 背景

V1.3b 已经实现已支付订单整单作废，并能自动回滚库存。现场售卖继续使用时，还需要处理更常见的售后记录问题：

- 为什么作废订单。
- 是否已经人工退款。
- 退款金额是多少。
- 退款通过什么方式完成。
- 退款是否影响销售统计。
- 订单列表中如何快速识别有售后处理的订单。

V1.3c 的目标是补齐“售后记录”能力，而不是接入真实支付平台退款。

## 2. 本版本目标

V1.3c 交付轻量、可靠、可追溯的订单售后记录能力：

- 订单作废时可记录作废原因。
- 已作废订单可查看作废原因和作废时间。
- 可为已支付或已作废订单记录人工退款。
- 退款记录只表达“线下人工处理已记录”，不自动调用微信/支付宝接口。
- 订单详情展示售后记录。
- 订单列表展示售后标识。
- 后续仪表盘可以基于这些数据计算净销售额、退款金额和作废订单数。

## 3. 本版本不包含

- 不接入微信退款 API。
- 不接入支付宝退款 API。
- 不做复合二维码。
- 不做复杂订单调整。
- 不做商品换货。
- 不做部分商品级退款明细。
- 不做订单调整表。
- 不做 V1.4 仪表盘。

## 4. 业务规则

### 4.1 订单状态

V1.3c 推荐继续沿用现有订单状态：

- `pending_payment`：待支付。
- `paid`：已支付。
- `cancelled`：已取消/已作废。

本版本暂不新增 `refunded` 或 `partially_refunded` 到订单主状态。

原因：

- 退款是售后记录，不一定改变订单原始支付事实。
- 一笔订单可以已支付后发生全额或部分人工退款。
- 如果直接把订单主状态改为 `refunded`，会让库存回滚、销售记录和退款记录混在一个状态字段里。

V1.3c 推荐新增订单售后记录，用记录表达退款，而不是用订单主状态承载所有含义。

### 4.2 作废原因

作废原因建议支持固定选项和可选备注。

固定选项：

- `mistake`：误操作。
- `customer_cancelled`：客户取消。
- `duplicate_order`：重复下单。
- `inventory_issue`：库存/赠品异常。
- `payment_issue`：收款异常。
- `other`：其他。

用户可见中文：

- 误操作。
- 客户取消。
- 重复下单。
- 库存/赠品异常。
- 收款异常。
- 其他。

规则：

- 点击 `作废订单` 后，确认弹窗增加作废原因选择。
- 默认为 `误操作`，降低现场操作成本。
- 备注可选，不作为必填项。
- 确认作废后，订单详情展示作废原因和备注。

### 4.3 人工退款记录

V1.3c 推荐新增“人工退款记录”能力。

记录字段建议：

- `id`：退款记录 ID。
- `orderId`：关联订单 ID。
- `amount`：退款金额。
- `method`：退款方式。
- `reason`：退款原因。
- `note`：备注。
- `createdAt`：记录时间。

退款方式建议：

- `wechat`：微信。
- `alipay`：支付宝。
- `cash`：现金。
- `other`：其他。

退款原因建议：

- `customer_return`：客户退单。
- `overcharge`：多收款。
- `product_issue`：商品问题。
- `manual_adjustment`：人工调整。
- `other`：其他。

规则：

- 已支付订单可以记录人工退款。
- 已作废订单也可以记录人工退款，用于记录作废后是否已向客户退回款项。
- 退款金额必须大于 0。
- 单笔订单累计退款金额不能超过订单应收金额。
- 退款记录不可直接删除，避免账目追溯缺失。
- 如果后续必须更正退款记录，应通过新增反向修正记录处理，V1.3c 暂不做。

### 4.4 销售统计口径

V1.3c 先定义口径，具体仪表盘可在 V1.4 实现。

建议统计口径：

- `订单原始销售额`：所有已支付订单的应收金额总和，不扣退款。
- `作废订单金额`：状态为 `cancelled` 的订单应收金额总和。
- `退款金额`：人工退款记录金额总和。
- `净销售额`：已支付订单应收金额 - 作废订单金额 - 人工退款金额。

注意：

- 如果已作废订单同时记录人工退款，净销售额不能重复扣减。
- V1.3c 需要在方案中明确：作废代表该订单销售额不计入净销售额；退款记录主要用于现金流追踪。
- V1.4 仪表盘实现时需要避免重复扣减。

## 5. 数据设计建议

### 5.1 Order 扩展字段

建议给 `Order` 增加可选字段：

- `cancelReason?: OrderCancelReason`
- `cancelNote?: string`

优点：

- 作废原因属于订单作废结果，直接放在订单上查询简单。
- 不需要为了一个原因字段新增单独表。

### 5.2 新增退款记录表

建议新增 `orderRefunds` 表。

原因：

- 一笔订单可能有多次人工退款记录。
- 退款是独立售后事件，不适合压扁在订单主表。
- 后续仪表盘统计退款金额会更清晰。

Dexie schema 需要升级。

建议索引：

- `id`
- `orderId`
- `createdAt`

### 5.3 备份影响

需要升级备份校验：

- 导出包含 `orderRefunds`。
- 导入校验退款金额、方式、原因、关联订单 ID。
- 允许旧备份不包含 `orderRefunds`，导入时按空数组处理。

是否升级备份版本号：

- 建议升级。
- 原因是新增表会改变备份结构，用户需要能区分 V1.3b 与 V1.3c 备份。

## 6. 页面与交互

### 6.1 订单详情

新增“售后记录”区域。

展示内容：

- 作废状态。
- 作废时间。
- 作废原因。
- 作废备注。
- 人工退款记录列表。
- 累计退款金额。

操作按钮：

- `记录退款`
- `作废订单`

显示规则：

- `paid`：显示 `作废订单` 和 `记录退款`。
- `cancelled`：不显示 `作废订单`，显示 `记录退款`。
- `pending_payment`：不显示售后操作。

### 6.2 作废确认弹窗

在现有确认弹窗中增加：

- 作废原因下拉菜单。
- 备注输入框。

确认按钮仍为：

- `确认作废`

失败提示仍必须是固定中文，不显示底层异常。

### 6.3 记录退款弹窗

字段：

- 退款金额。
- 退款方式。
- 退款原因。
- 备注。

校验：

- 金额必填。
- 金额必须大于 0。
- 累计退款金额不能超过订单应收金额。
- 校验失败在弹窗内显示中文红字。

保存后：

- 关闭弹窗。
- 订单详情刷新售后记录。
- 售卖页显示成功提示：`订单 订单号 已记录退款 ¥退款金额。`

### 6.4 订单列表

订单记录行建议增加售后标识：

- 已作废。
- 已退款。
- 部分退款。

建议保持紧凑展示，避免订单列表高度增加太多。

## 7. Repository 设计

建议新增函数：

- `voidPaidOrder(orderId, options, now)`
  - `options.cancelReason`
  - `options.cancelNote`
- `saveOrderRefund(refundInput, now)`
- `listOrderRefunds(orderId)`
- `listRefunds()`

事务规则：

- 作废订单仍然在单事务中完成订单状态、库存回滚、库存流水写入。
- 记录退款不改库存。
- 记录退款需要校验累计退款金额不能超过订单应收金额。

## 8. 测试策略

### 8.1 Repository 测试

覆盖：

- 作废订单保存作废原因和备注。
- 旧调用不传原因时使用默认原因。
- 记录单笔退款成功。
- 多笔退款累计不能超过订单应收金额。
- 非法金额被拒绝。
- 待支付订单不能记录退款。
- 查询订单退款记录按时间排序。

### 8.2 组件测试

覆盖：

- 作废确认弹窗显示原因下拉。
- 记录退款弹窗校验必填金额。
- 订单详情显示售后记录。
- 已作废订单不显示作废按钮但显示记录退款入口。

### 8.3 页面测试

覆盖：

- 从订单详情作废并保存原因。
- 从订单详情记录退款。
- 退款成功后刷新订单详情。
- 失败时显示固定中文错误。
- 不显示原始内部错误。

### 8.4 备份测试

覆盖：

- 新备份导出包含退款记录。
- 新备份导入恢复退款记录。
- 旧备份不包含退款记录时仍可导入。
- 非法退款记录导入失败。

## 9. 验收标准

- 作废订单时可以选择原因。
- 作废后订单详情展示作废原因和作废时间。
- 已支付订单可以记录人工退款。
- 已作废订单可以记录人工退款。
- 退款金额不能超过订单应收金额。
- 订单详情可看到退款记录和累计退款金额。
- 订单列表能识别已作废、已退款或部分退款订单。
- 导出/导入备份后售后记录不丢失。
- 全量测试通过。
- 生产构建通过。

## 10. 推荐实施拆分

### 10.1 V1.3c-1：作废原因

- 扩展订单类型。
- 修改作废确认弹窗。
- 修改 repository。
- 修改备份校验。
- 增加订单详情展示。

### 10.2 V1.3c-2：人工退款记录

- 新增 Dexie 表。
- 新增退款 repository。
- 新增退款弹窗。
- 新增订单详情退款记录展示。
- 新增备份导入导出。

### 10.3 V1.3c-3：订单列表售后标识

- 订单列表展示已作废、已退款、部分退款。
- 优化筛选和搜索。
- 为 V1.4 仪表盘准备统计口径。

## 11. 推荐结论

V1.3c 推荐先从 `作废原因` 开始，再做 `人工退款记录`，最后做 `订单列表售后标识`。

原因：

- 作废原因改动最小，能延续 V1.3b 的订单作废能力。
- 人工退款需要新增表和备份版本，风险更高，应单独推进。
- 订单列表标识依赖退款记录和作废原因完成后再统一设计。
