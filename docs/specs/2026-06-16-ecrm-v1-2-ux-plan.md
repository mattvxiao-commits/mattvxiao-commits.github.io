# ECRM V1.2 核心操作流 UE 重构方案

## 版本定位

V1.2 聚焦“现场高频操作链路”的可用性与效率，不新增复杂经营分析模块，不处理 GitHub 部署和多设备发布教学，不实现 P3 的复合二维码与飞书式复杂仪表盘。

本版本目标是把当前已能跑通的商品管理、售卖、收款流程进一步整理成更适合线下摆摊现场使用的操作体验：

- 商品多时，新增、编辑、复制商品不需要来回滚动页面。
- 大量相似 SKU 可以通过复制创建，减少重复录入。
- 进入收款后，页面默认服务于“订单复核 + 收款确认”，不再继续展示全部售卖商品造成干扰。
- 收款方式与二维码展示一一对应，减少右侧区域高度浪费。
- 赠品选择区文案更清楚，明确展示“赠品A / 赠品B”和具体 SPU/SKU。

## 当前问题

### 商品管理

当前商品新增和编辑使用页面顶部展开式表单。商品数量较少时可用，但商品列表较长后存在明显问题：

- 在列表中部或底部点击“编辑”后，用户当前位置没有即时反馈。
- 编辑表单展开在页面顶部，用户必须手动滚回顶部。
- 操作员容易误以为点击无效。
- 新增相似 SKU 时只能从空表单开始填写，重复录入成本高。

### 收款确认

当前点击“去收款”后，左侧仍展示全部可售商品列表，右侧展示收款确认。该布局在现场收款阶段目标不够集中：

- 收款阶段核心任务应是复核订单、确认金额、选择收款方式、选择赠品 SKU、保存订单。
- 左侧继续展示全部商品容易让用户误以为仍处于选购阶段。
- 商品列表和收款确认并列后，页面信息量过大，空间利用率不佳。
- 微信/支付宝二维码同时显示，占用大量高度；实际同一时间只需展示当前选择的收款方式。
- 右下角购物车浮条在 checkout 模式下仍显示，可能遮挡收款区域。

## V1.2 范围

### P0：必须完成

1. 商品新增/编辑改为大弹窗
   - 点击“新增商品”直接打开弹窗。
   - 点击任意商品“编辑”直接打开弹窗。
   - 不再在页面顶部展开商品表单。
   - 弹窗内复用现有 ProductForm 字段、校验、图片上传和编辑保存确认逻辑。
   - 弹窗内部可滚动，底部操作区固定，避免长表单导致按钮不可见。

2. 商品复制创建相似 SKU
   - 商品列表增加“复制”操作。
   - 点击复制后打开新增商品弹窗，并带入原商品大部分信息。
   - 复制时保留：商品名称、SPU、SPU 编码、成本价、售价、商品图、可售卖、可作为赠品、状态。
   - 复制时清空：SKU 编码、完整商品编码。
   - 复制时库存默认为 0，避免误把原库存复制成新 SKU 可售库存。
   - 保存时继续使用完整商品编码重复校验。

3. 收款页左侧改为订单商品复核区
   - checkout 模式下，左侧不再默认显示全部售卖商品。
   - 左侧显示当前购物车内商品列表。
   - 每行展示商品小图、商品名、SPU、单价、数量、小计、优惠类型。
   - 左侧底部显示件数、原价小计、优惠金额、应收金额。
   - 需要继续添加商品时，点击“返回”回到售卖商品列表。

4. 收款二维码按支付方式单独显示
   - 选择微信：只显示微信收款码。
   - 选择支付宝：只显示支付宝收款码。
   - 选择现金：不显示二维码，显示现金收款确认提示。
   - 选择其他：不显示二维码，显示其他收款方式确认提示。
   - 当前方式二维码未设置时，只显示该方式的未设置状态。

5. checkout 模式空间压缩
   - checkout 模式隐藏右下角购物车浮条。
   - checkout 模式压缩顶部标题区和无关控件。
   - checkout 模式不展示 SPU 筛选和商品展示方式切换。
   - 目标是在桌面横屏中尽量一屏显示订单复核、金额、收款方式、当前二维码、赠品选择和确认按钮。

6. 赠品选择区文案优化
   - SPU 赠品选择区标题显示为“赠品A：xxx”或“赠品B：xxx”。
   - 文案保留“需选 n 个，已选 m 个”。
   - 下拉选项继续显示商品编码、商品名、库存；无编码商品不显示“未设置编码”。

### P1：本版本尽量完成

1. 商品弹窗关闭保护
   - 当前表单有未保存修改时，点击取消或遮罩关闭前提示确认。
   - 若实现成本过高，可先仅保留取消按钮关闭，不支持点击遮罩关闭。

2. 订单复核区视觉压缩
   - 商品行比购物车抽屉略宽松，便于收款前复核。
   - 商品多时左侧列表自身滚动，不让整页过度下拉。

### 本版本不做

- 不做飞书式可配置仪表盘。
- 不做复合二维码。
- 不做云同步。
- 不做 GitHub 部署教学。
- 不做原生 iPad App。
- 不做多购物车暂存。
- 不做订单详情、退款、作废、库存流水，这些留到 V1.3。

## 交互设计

### 商品弹窗

桌面端：

- 弹窗居中，宽度约 760 至 880px。
- 最大高度为视口高度减去安全边距。
- 表单内容区域滚动。
- 弹窗底部固定展示“取消 / 删除 / 保存商品”。

iPad 和窄屏：

- 弹窗宽度接近全屏。
- 图片上传区域与字段区上下排列。
- 按钮不遮挡表单内容。

复制创建：

- 列表商品操作区显示“编辑 / 复制”。
- 点击“复制”打开新增商品弹窗。
- 弹窗标题显示“复制创建商品”。
- 保存后作为新商品插入列表。

### 收款确认页

checkout 模式布局：

- 左侧：订单商品复核区。
- 右侧：收款确认区。

左侧订单复核区：

- 标题：订单商品。
- 顶部显示“共 n 件 / 应收 ¥x”。
- 商品行展示：缩略图、商品名、SPU、单价、数量、小计、优惠标签。
- 底部显示：原价小计、优惠、应收。

右侧收款确认区：

- 顶部：金额和返回按钮。
- 收款方式：微信、支付宝、现金、其他。
- 二维码区：仅显示当前选择方式。
- 赠品选择区：显示赠品A/B、SPU/SKU、需选数量、已选数量。
- 底部：确认已收款并保存订单。

## 数据与逻辑影响

### 商品复制

复制不直接写数据库，只生成 ProductForm 初始草稿。保存时仍走现有新增商品流程。

新商品必须生成新的完整商品编码：

- 原商品 SPU 编码保留。
- SKU 编码清空。
- 用户填写 SKU 编码后生成完整商品编码。
- 若与已有商品重复，沿用现有重复编码错误提示。

### 收款页订单复核

订单复核区使用现有 `calculated.lines` 展示，不重新计算价格。

赠品选择仍使用现有 `giftSelection.ts` 逻辑：

- `buildGiftSelectionRequirements`
- `validateGiftSelections`
- `resolveGiftLines`

V1.2 只调整展示与文案，不改变满赠规则、不改变库存扣减逻辑。

### 二维码展示

二维码仍使用设置页上传的微信/支付宝图片。现金和其他不需要二维码图片。

## 验收标准

### 商品管理

- 在商品列表任意滚动位置点击编辑，应立即出现编辑弹窗。
- 编辑弹窗保存后列表更新。
- 编辑弹窗删除商品仍可用。
- 点击新增商品应打开新增弹窗。
- 点击复制应打开复制创建弹窗。
- 复制创建时 SKU 编码为空，库存为 0。
- 复制创建保存时若编码重复，应显示现有重复编码错误。
- 商品图片在复制草稿中可继承并显示。

### 收款确认

- 点击去收款后，左侧显示购物车商品，不显示全部售卖商品。
- checkout 模式不显示 SPU 筛选、不显示商品视图切换。
- checkout 模式不显示右下角购物车浮条。
- 微信/支付宝/现金/其他切换时，二维码区只显示当前方式对应内容。
- 现金和其他方式不显示微信/支付宝二维码卡片。
- SPU 赠品选择标题显示“赠品A：xxx”或“赠品B：xxx”。
- 确认收款前仍会校验赠品 SKU 是否选择完整。
- 成功保存订单后仍清空购物车并扣减库存。

## 测试策略

### 自动测试

1. Product management
   - 新增商品按钮打开弹窗。
   - 编辑商品按钮打开弹窗，不依赖页面顶部表单。
   - 复制商品按钮打开新增弹窗，并带入预期字段、清空 SKU、库存为 0。
   - 保存复制商品时调用新增逻辑。

2. Checkout layout
   - checkout 模式下左侧展示购物车商品。
   - checkout 模式下不展示全部商品列表。
   - checkout 模式下隐藏购物车浮条。
   - 支付方式切换只显示当前二维码状态。
   - 现金/其他显示文字确认状态。
   - SPU 赠品选择标题包含赠品A/B。

3. Regression
   - `npm test`
   - `npm run build`

### 手工验收

- 桌面横屏 100% 宽度。
- 桌面窗口 60% 至 80% 宽度。
- iPad 横屏。
- iPad 竖屏。

重点观察：

- 弹窗是否遮挡、是否能滚动、按钮是否可见。
- checkout 模式是否能一屏看到主要信息。
- 赠品选择是否清楚。
- 二维码切换是否符合直觉。
