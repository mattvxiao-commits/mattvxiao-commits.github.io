# ECRM V1.3b 订单作废方案

## 1. 目标

V1.3b 交付“已支付订单整单作废”能力，用于现场售卖时处理误保存、重复保存、客户临时取消付款后已误点确认等情况。

本版本必须真实修改订单状态并回滚库存，不只是 UI 标记。作废后订单仍保留在订单记录中，便于对账和追溯。

## 2. 范围

### 2.1 本版本包含

- 在订单详情中对 `paid` 订单显示“作废订单”按钮。
- 点击后弹出二次确认弹窗。
- 确认作废后：
  - 订单状态从 `paid` 改为 `cancelled`。
  - 写入 `cancelledAt`。
  - 按该订单原库存扣减流水反向回滚库存。
  - 写入库存回滚流水。
  - 刷新订单列表与订单详情。
- 作废后的订单在订单详情中显示为“已取消”。
- 作废后的订单不再显示“作废订单”按钮。
- 作废失败时显示中文错误提示。

### 2.2 本版本不包含

- 不做部分退款。
- 不做全额退款。
- 不做订单调整表。
- 不做订单调整明细表。
- 不新增 Dexie schema 版本。
- 不升级备份版本。
- 不做复杂原因分类和备注字段。
- 不把 `cancelled` 改名为 `voided`。

## 3. 业务规则

### 3.1 状态规则

当前数据模型已经有：

- `pending_payment`
- `paid`
- `cancelled`

V1.3b 沿用 `cancelled` 表示“已取消/已作废”。原因是当前 IndexedDB schema 已支持该状态，备份校验也已支持该状态，可以避免为了一个紧急小版本引入 schema 和备份版本升级。

后续 V1.3c/V1.3d 如果引入退款和订单调整表，再统一评估是否升级为更细的 `voided` / `refunded` 状态。

### 3.2 可作废条件

只有 `paid` 订单可以作废。

以下订单不能作废：

- `cancelled` 订单。
- `pending_payment` 订单。
- 找不到订单的情况。
- 找不到对应库存扣减流水的情况。

不能重复作废同一订单。

### 3.3 库存回滚规则

作废使用该订单已经写入的库存流水作为依据，而不是重新计算购物车或读取当前促销规则。

对每一条原始扣减流水：

- 原流水 `changeQty` 为负数。
- 回滚流水 `changeQty` 为其绝对值。
- 回滚前库存为商品当前库存。
- 回滚后库存为当前库存 + 回滚数量。
- 商品 `updatedAt` 使用作废时间。

示例：

- 原流水：普通商品库存 `10 -> 8`，`changeQty = -2`。
- 当前库存：`6`。
- 作废回滚：`6 -> 8`，`changeQty = +2`。

这样可以正确处理订单之后又发生其他销售的情况。

### 3.4 库存流水原因

V1.3b 新增库存流水原因：

- `order_cancelled_rollback`：订单作废回滚。

原因映射：

- `order_paid`：订单扣减。
- `gift_order_paid`：赠品扣减。
- `order_cancelled_rollback`：作废回滚。
- `manual_adjust`：手动调整。

### 3.5 失败规则

作废必须在事务中完成。

如果过程中出现以下情况，不能部分写入：

- 订单不存在。
- 订单不是 `paid`。
- 商品不存在。
- 原始库存扣减流水不存在。
- 原始流水不是负数扣减。
- 数据库写入失败。

用户可见错误提示必须为中文，不把底层异常栈或内部路径暴露到 UI。

## 4. 交互设计

### 4.1 订单详情

在订单详情弹窗底部增加操作区：

- `关闭`
- `作废订单`

显示规则：

- `paid`：显示“作废订单”。
- `cancelled`：不显示“作废订单”。
- `pending_payment`：不显示“作废订单”。

### 4.2 作废确认

点击“作废订单”后显示确认弹窗，文案：

- 标题：`确认作废订单`
- 风险提示：`作废后订单会标记为已取消，并自动回滚本订单扣减的库存。此操作不可撤销。`
- 订单号：展示当前订单号。
- 按钮：
  - `取消`
  - `确认作废`

V1.3b 暂不要求填写作废原因，原因字段留到 V1.3d 订单调整记录中处理。

### 4.3 成功与失败反馈

作废成功：

- 关闭确认弹窗。
- 订单详情更新为已取消。
- 库存流水摘要出现“作废回滚”记录。
- 售卖页顶部显示成功提示：`订单 ECRM-xxx 已作废，库存已回滚。`
- 订单列表刷新后仍可通过状态筛选查看已取消订单。

作废失败：

- 确认弹窗保持或关闭均可，优先保持当前详情弹窗。
- 售卖页显示：`订单作废失败，请刷新后重试。`

## 5. 数据与备份影响

### 5.1 数据模型

修改 `InventoryLog.reason` 类型，增加 `order_cancelled_rollback`。

不新增表。

不修改 `Order` 字段。

### 5.2 Dexie

不升级 Dexie schema。

原因：

- `orders` 表已经包含 `status`、`cancelledAt` 字段可以自然存储。
- `inventoryLogs` 表已有 `orderId`、`createdAt` 索引，可以查询同订单流水。
- 新的 `reason` 是普通字段值变化，不需要索引。

### 5.3 备份

备份版本不升级。

需要同步更新备份校验允许的库存流水原因：

- 增加 `order_cancelled_rollback`。

否则作废后的数据导出再导入会被校验拒绝。

## 6. 测试策略

### 6.1 Repository 测试

覆盖：

- 作废 paid 订单后，订单状态变为 `cancelled`，写入 `cancelledAt`。
- 商品库存按当前库存回滚。
- 写入 `order_cancelled_rollback` 库存流水。
- 重复作废被拒绝。
- 非 paid 订单作废被拒绝。
- 原始扣减流水缺失时拒绝。
- 任一失败不产生部分写入。

### 6.2 组件测试

覆盖：

- `paid` 订单详情显示“作废订单”。
- `cancelled` 订单详情不显示“作废订单”。
- 点击作废按钮后展示确认弹窗。
- 点击确认调用上层 `onVoidOrder`。
- 不显示退款入口。

### 6.3 页面测试

覆盖：

- 售卖页打开订单详情后作废订单。
- 调用 repository 的 `voidPaidOrder(order.id)`。
- 成功后刷新商品、订单、订单明细、库存流水。
- 成功提示为中文。
- 失败提示为中文且不泄露底层错误。

## 7. 验收标准

- 已支付订单详情中可作废。
- 作废前有二次确认。
- 作废后订单状态为已取消。
- 作废后商品库存自动增加回原订单扣减数量。
- 作废后库存流水中能看到作废回滚记录。
- 作废订单不能再次作废。
- 订单列表可通过状态筛选查看已取消订单。
- 全量测试和生产构建通过。
