# ECRM 后续开发规划与待办清单

日期：2026-06-25  
当前开发分支：`v1.7-hybrid-full-prototype`  
当前线上 / 远端可见版本：仍按 `v1.6.2` 识别  
当前开发状态：已进入 `v1.7.x` 本地开发阶段，尚未做 `v1.7.0` 正式发布

## 1. 文档目的

本文用于记录 ECRM 后续开发优先级、阶段安排和待办事项，避免后续迭代中遗漏已经讨论确认过的内容。

当前判断是：先把 `V1.7` 桌面 / 浏览器主使用场景收稳，再做一次正式发布；随后进入运营活动与促销规则模块化。移动端需要专项处理，但不应在活动规则和桌面端结构尚未稳定时提前大改。

## 2. 当前版本判断

### 2.1 已完成但尚未正式发布的 V1.7 内容

当前本地分支已经完成多项 `V1.7` 相关能力：

- 桌面端左侧窄导航 App Shell。
- 独立订单页。
- 订单页双列订单列表。
- 订单列表按生成顺序两两成组，最新一组在最上方，组内左旧右新。
- 商品新增 / 编辑中的 `系列（筛选）` 字段。
- 售卖页系列筛选与 SPU 筛选联动。
- 设置页现场模式 `锁定范围` 多选。
- 现场模式入口三态识别：未锁定、临时解锁、现场模式。
- 窄屏 / 移动端基础底部导航壳层。

这些内容仍属于本地开发成果。正式发布前需要完成剩余桌面页面级适配和版本发布动作。

### 2.2 当前不应误判为已经完成的内容

以下内容还不能视为完成：

- `v1.7.0` 正式发布。
- 远端仓库版本号、README、tag、GitHub Pages 部署同步。
- 其他设备安装 / 更新验收。
- 移动端专项适配。
- 运营活动 / 促销规则模块化。
- 全局 UI/UE 规格统一审查。

当前移动端底部导航只属于基础壳层适配，不能等同于完整移动端产品体验。

## 3. 总体推进顺序

推荐推进次序如下：

1. `V1.7` 桌面 / 浏览器页面级适配收尾。
2. `v1.7.0` 正式发布，远端同步，其他设备安装更新测试。
3. `V1.8` 运营活动 / 促销规则模块化。
4. `V1.8.x` 购物车、订单、仪表盘、导出、备份对活动模块的联动适配。
5. 高优先级待办补齐。
6. 桌面端整体 UI/UE 规格统一审查。
7. 移动端专项适配。

这条顺序的原因很明确：如果过早做移动端专项或全局 UI 精修，后续运营活动模块一上线，购物车、设置页、仪表盘和订单详情都会再次变化，返工成本会很高。

## 4. P0：完成 V1.7 桌面 / 浏览器适配

### 4.1 阶段目标

让 `V1.7` 在 Windows / Mac 浏览器、iPad 横屏、窄桌面窗口下具备稳定可用的桌面体验。

本阶段不做全局 UI/UE 细节大审，也不做移动端专项重构。只处理新 App Shell 下明显影响使用的页面级问题。

### 4.2 待处理页面

商品页：

- 检查左侧壳层下标题区、操作区、商品列表是否稳定。
- 检查新增 / 编辑商品弹窗是否仍完整保留字段。
- 检查复制创建相似 SKU 是否仍可用。
- 检查系列字段、SPU/SKU 编码和完整商品编码是否显示清楚。

售卖页：

- 检查系列筛选、SPU 筛选和商品列表在新壳层下是否稳定。
- 检查购物车收缩 / 展开是否受左侧壳层影响。
- 检查购物车明细、赠礼 / 出库按钮、备注、促销摘要、金额区域是否完整。
- 检查加购、满赠、运营赠礼、人工赠送、其他出库在购物车内的展示是否清楚。

收款页：

- 检查收款页在新壳层下是否仍以订单复核和付款为主。
- 检查二维码、收款方式、金额、赠品库存提示、确认收款按钮是否无缺失。
- 检查商品数量较多时，商品清单和金额区是否不会互相遮挡。

订单页：

- 检查独立订单页的筛选、排序、订单详情入口是否稳定。
- 检查支付方式标签颜色区分是否清楚。
- 检查订单性质标签是否能正确反映整单 / 单商品统计口径修正后的状态。
- 检查订单详情弹窗是否完整保留正式版内容。

数据页：

- 检查时间范围筛选、统计口径筛选在新壳层下是否稳定。
- 检查经营概览、销售、活动、毛利、库存、异常等分区是否完整。
- 检查窄桌面窗口下顶部筛选区是否不会挤压失控。

设置页：

- 检查现场模式、基础信息、收款码、促销配置、运营赠礼、备份恢复、订单 Excel、系统信息是否完整。
- 检查现场模式 PIN、锁定范围、立即重新锁定、关闭现场模式是否可用。
- 检查导入备份、导出备份、导出订单 Excel 是否不受新壳层影响。

### 4.3 验收标准

- 桌面宽屏下五个主页面均可稳定使用。
- 窄桌面窗口下不会出现导航、筛选区、按钮区域严重挤压。
- iPad 横屏接近桌面体验。
- 购物车、收款页、订单详情没有功能缺失。
- 订单记录、退款、作废、统计口径修正仍可用。
- 仪表盘统计内容不因 UI 改版缺失。
- 设置页导入导出、Excel 导出、现场模式仍可用。

### 4.4 验证命令

每次小版本完成后至少运行：

```text
npm test -- --run
npm run build
git diff --check
```

如涉及页面结构变化，应补充本地人工验收：

- 桌面宽屏。
- 60%-80% 浏览器宽度。
- iPad 横屏或等效窗口尺寸。
- 窄屏基础可用性检查。

## 5. P0：v1.7.0 正式发布与远端同步

### 5.1 触发条件

只有当 `V1.7` 桌面 / 浏览器适配收尾完成并通过本地验收后，才进入正式发布。

### 5.2 推荐版本号

建议发布版本号为：

```text
v1.7.0
```

原因：

- 当前本地开发已经明显超过 `v1.6.2`。
- `V1.7` 包含 App Shell、订单页、系列筛选、现场模式范围、移动端基础壳层等结构性变化。
- 使用 `v1.7.0` 便于后续用 `v1.7.1`、`v1.7.2` 做热修。

### 5.3 发布待办

发布前：

- 更新 `package.json` 版本号。
- 确认页面顶部和设置页系统信息显示 `v1.7.0`。
- 更新 README 中的版本说明和功能说明。
- 更新必要的发布记录文档。
- 全量测试通过。
- 生产构建通过。
- 本地预览通过。

远端同步：

- 合并当前开发分支到 `main`。
- 推送 `main` 到 GitHub 远端。
- 创建 `v1.7.0` tag。
- 推送 tag。
- 等待 GitHub Actions 部署完成。
- 打开 GitHub Pages 地址确认线上版本更新。

其他设备验收：

- Windows / Mac 浏览器打开线上地址。
- iPad 打开线上地址。
- PWA 添加到主屏幕或重新打开已安装应用。
- 检查版本号显示。
- 检查旧数据是否仍在本机 IndexedDB 中。
- 使用 JSON 备份在其他设备导入测试。
- 检查商品图、订单、库存、设置是否恢复正常。

### 5.4 发布风险

- PWA 缓存可能导致旧版本暂时可见。
- 旧设备可能需要刷新、关闭浏览器或重新打开 PWA 才能看到新版本。
- GitHub Actions 部署有延迟。
- 线上版本切换后，需要确认当前数据仍保存在各设备本地，不会自动跨设备同步。

## 6. P1：V1.8 运营活动 / 促销规则模块化

### 6.1 阶段目标

把当前写死的加购、满赠、运营赠礼能力，升级为可配置、可扩展的运营活动模块。

当前系统里的促销能力更接近一次具体摆摊活动的定制规则。发布给更多用户时，必须允许用户按自己的摊位规则配置活动。

### 6.2 设计原则

不建议直接做自由公式编辑器。自由度太高会增加学习成本，也容易配置出冲突规则。

推荐采用：

```text
模板化活动 + 统一规则引擎
```

用户看到的是简单模板：

- 满额赠品。
- 加购优惠。
- 折扣优惠。
- 运营赠礼。

系统内部使用统一结构计算活动命中、价格、赠品、成本和订单快照。

### 6.3 统一活动模型

建议底层统一为 `Activity` 模型，包含：

- 活动 ID。
- 活动名称。
- 活动类型。
- 是否启用。
- 生效时间，可先做可选。
- 触发条件。
- 作用范围。
- 优惠结果。
- 限制条件。
- 叠加规则。
- 成本归属。
- 优先级。
- 订单快照字段。

简化结构示意：

```text
Activity
- id
- name
- type
- enabled
- condition
- scope
- benefit
- limit
- stackPolicy
- accountingCategory
- priority
```

### 6.4 活动模板一：满额赠品

用户可配置：

- 活动名称。
- 是否启用。
- 满多少金额触发。
- 赠送 SKU 或 SPU。
- 每档赠送数量。
- 多档是否叠加，默认只取最高档。
- 赠品库存不足时如何提示。
- 成本归属，默认计入满赠成本或运营活动成本。

需要替代当前写死的三档规则：

- 满 35。
- 满 68。
- 满 148。

未来用户可以自行新增、删除、调整档位。

### 6.5 活动模板二：加购优惠

用户可配置：

- 活动名称。
- 是否启用。
- 触发条件：任意消费后、满指定金额后、购买指定 SKU/SPU/系列后。
- 加购范围：指定 SKU、指定 SPU、指定系列。
- 原价。
- 优惠价。
- 每单最多优惠件数。
- 超出数量是否恢复原价。
- 是否允许同一加购 SPU 的首件原价、后续优惠。

需要覆盖当前规则：

- 任意消费后，加购某 SPU 前 3 个享受优惠价。
- 如果只买该 SPU，则第 1 个原价，第 2-4 个优惠，第 5 个恢复原价。

### 6.6 活动模板三：折扣优惠

用户可配置：

- 活动名称。
- 是否启用。
- 触发条件：满额、指定 SKU、指定 SPU、指定系列、全场。
- 折扣方式：百分比折扣、立减金额、指定优惠价。
- 作用范围：整单、指定 SKU、指定 SPU、指定系列。
- 是否可与满赠、加购叠加。
- 优先级。

折扣优惠不建议第一版做得过于复杂。第一版优先支持线下摊位最常见的简单折扣：

- 满额整单折扣。
- 指定 SPU 折扣。
- 指定系列折扣。

### 6.7 活动模板四：运营赠礼

用户可配置：

- 活动名称。
- 是否启用。
- 是否要求购物车内有正常销售商品。
- 赠礼范围：可赠品 SKU、可赠品 SPU。
- 默认赠品。
- 备注模板。
- 成本归属，默认计入运营活动成本。

典型场景：

- 关注社媒赠礼。
- 加入社群赠礼。
- 现场互动赠礼。
- 打卡赠礼。

这类赠礼默认不计销售收入，但要扣库存、进订单明细、进库存流水，并进入仪表盘活动成本统计。

### 6.8 人工赠送 / 其他非销售出库

这两类不应归入官方运营活动。

它们用于处理：

- 好友赠送。
- 补偿赠送。
- 商品损耗。
- 展示样品消耗。
- 其他非销售出库。

默认规则：

- 不计销售收入。
- 不计销售毛利。
- 扣库存。
- 记录库存流水。
- 计入非经营出库成本。
- 可在仪表盘中单独筛选。

## 7. P1：V1.8.x 活动模块联动

运营活动模块不是设置页加几个输入框。它会影响核心业务链路，必须分小步推进。

### 7.1 购物车联动

待办：

- 使用新规则引擎计算购物车活动命中。
- 清楚展示活动命中结果。
- 显示满赠、加购、折扣、运营赠礼的来源。
- 避免多个活动同时命中时展示混乱。
- 处理活动叠加和优先级。

### 7.2 订单联动

待办：

- 订单保存活动快照。
- 订单明细保存活动来源。
- 订单详情显示活动命中信息。
- 统计口径修正不应破坏原始活动快照。
- 退款、作废时保留原活动记录。

### 7.3 仪表盘联动

待办：

- 支持按活动查看销售额、赠品数量、活动成本。
- 支持区分销售成本、满赠成本、运营赠礼成本、非经营出库成本。
- 支持筛选正常销售、运营赠礼、人工赠送、其他出库。
- 支持查看活动效果，例如活动带来的订单数、赠品消耗、活动成本占比。

### 7.4 导出联动

待办：

- 订单 Excel 增加活动相关字段。
- 订单明细导出活动类型、活动名称、成本归属。
- 导出说明中解释活动成本和非销售出库口径。

### 7.5 备份联动

待办：

- JSON 备份包含活动配置。
- 导入旧备份时生成兼容默认活动配置。
- 导入新备份时恢复活动模板、规则、订单快照。
- 保持旧版数据可读。

- 活动规则排序。
- 活动命中说明优化。
- 商品批量管理。
- 库存入库记录。

- 商品损耗记录。

这些事项需要按实际使用反馈再排序。

## 9. P2：桌面端整体 UI/UE 规格统一审查

### 9.1 触发条件

建议在 `V1.8` 活动模块完成并通过基础验收后，再做这一轮。

原因：

- 活动模块会改变购物车、设置页、订单详情、仪表盘。
- 如果先做 UI 精修，后面很可能再次返工。

### 9.2 审查范围

- 页面标题区。
- 顶部筛选区。
- 页面主操作区。
- 表单字段规格。
- 按钮尺寸和层级。
- 标签 / chip 规格。
- 弹窗规格。
- 抽屉规格。
- 订单列表规格。
- 仪表盘模块规格。
- 商品、售卖、订单、数据、设置之间的密度一致性。

### 9.3 验收目标

- 桌面宽屏空间利用率合理。
- 窄桌面窗口不明显失控。
- 页面之间的标题、按钮、筛选、标签视觉规格统一。
- 高频操作路径清楚。
- 低频配置不抢占高频操作空间。

## 10. P3：移动端专项适配

### 10.1 当前判断

移动端不能只靠桌面页面自动缩放解决。

当前已完成的底部导航只解决基础壳层问题，目标是避免页面在窄屏下直接坏掉。它不是完整移动端方案。

### 10.2 移动端专项目标

手机竖屏：

- 可完成开单。
- 可添加购物车。
- 可确认收款。
- 可查询订单。
- 可查看关键库存和经营数据。

iPad 横屏：

- 尽量接近桌面体验。
- 保持商品列表、购物车、收款流程高效。

iPad 竖屏 / 大屏手机：

- 采用介于桌面和手机之间的布局。
- 避免简单压缩桌面布局。

### 10.3 页面专项方向

商品页：

- 移动端适合管理卡片或紧凑列表。
- 新增 / 编辑应优先使用全屏弹层。
- 批量录入和复制 SKU 仍要可用。

售卖页：

- 优先保证商品选择、购物车、收款三步流程。
- 商品筛选区需要压缩或折叠。
- 购物车更适合作为底部抽屉或全屏子页面。

收款页：

- 二维码和金额优先。
- 商品复核区应可折叠或放到二维码下方。
- 收款确认按钮需要固定在安全可触达区域。

订单页：

- 订单列表单列。
- 订单详情分区折叠。
- 售后、库存流水、活动信息不应一次性全部展开。

数据页：

- 以阅读复盘为主。
- 分区导航需要更明显。
- 模块说明入口需要补齐。

设置页：

- 高频设置靠前。
- 复杂配置使用分组折叠。
- 活动规则编辑应分步骤完成。

### 10.4 建议时机

移动端专项建议放在活动模块和桌面 UI/UE 规格统一之后。

原因：

- 那时页面内容更稳定。
- 活动配置结构更稳定。
- 仪表盘指标更稳定。
